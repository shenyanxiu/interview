# 后端性能优化有哪些方向

## 问题

后端性能优化有哪些方向，从请求链路来分析。

## 主题

性能优化

## 难度

中级

## 参考答案

### 1. 接入层

**负载均衡**
- 多实例部署 + 负载均衡（Nginx、ALB）
- 合理的分发策略：轮询、加权、最少连接、一致性哈希
- 健康检查剔除故障节点

**连接管理**
- Keep-Alive 复用连接，减少 TCP 握手开销
- 连接池管理（数据库、Redis、HTTP 客户端）
- 限流熔断（令牌桶、滑动窗口），防止雪崩

**协议优化**
- HTTP/2 减少头部开销
- gRPC 用于内部服务通信（二进制协议、多路复用）
- WebSocket 替代轮询场景

### 2. 应用层

**代码与逻辑优化**
- 避免 N+1 查询，批量操作替代循环单条
- 异步化：非关键路径走消息队列（发邮件、写日志、生成报表）
- 并行化：无依赖的 IO 操作并发执行（Promise.all / CompletableFuture）
- 减少序列化/反序列化开销（选择高效格式：protobuf > JSON > XML）

**缓存策略（最核心的优化手段）**
- 多级缓存：本地缓存（Caffeine/LRU）→ 分布式缓存（Redis）→ DB
- 缓存模式：Cache-Aside、Read/Write-Through、Write-Behind
- 热点数据预加载，冷数据懒加载
- 注意缓存穿透、击穿、雪崩的防护

**计算优化**
- 预计算：定时任务提前算好结果（排行榜、统计报表）
- 空间换时间：冗余字段避免 JOIN、物化视图
- 算法优化：合适的数据结构、避免不必要的深拷贝

### 3. 数据层

**数据库优化**
- 索引：覆盖索引、联合索引最左匹配、避免索引失效
- SQL 优化：EXPLAIN 分析执行计划、避免全表扫描、合理分页
- 读写分离：主库写、从库读
- 分库分表：水平拆分应对数据量增长

**查询模式优化**
- 避免 SELECT *，只查需要的字段
- 大事务拆小事务，减少锁持有时间
- 批量写入替代逐条插入
- 慢查询监控与治理

**存储选型**
- 关系型 vs NoSQL 按场景选择
- 时序数据用 InfluxDB/TimescaleDB
- 搜索场景用 Elasticsearch
- 高频读写用 Redis，持久化用 DB

### 4. 架构层

**服务拆分**
- 微服务化：独立扩缩容，隔离故障域
- 核心链路与非核心链路分离
- BFF 层聚合，减少前端请求次数

**弹性伸缩**
- 容器化 + K8s 自动扩缩容（HPA）
- 按 CPU/内存/QPS 指标弹性伸缩
- Serverless 应对突发流量

**异步解耦**
- 消息队列削峰填谷（Kafka、RabbitMQ）
- 事件驱动架构，减少同步等待
- CQRS：读写模型分离，各自优化

### 5. 基础设施层

**网络**
- CDN 加速静态资源和 API 响应缓存
- 就近部署，多区域多可用区
- 内网通信走专线，减少公网延迟

**系统调优**
- JVM 调优：GC 策略选择、堆大小、线程池配置
- Node.js：Cluster 模式利用多核、合理的事件循环管理
- Linux 内核参数：TCP 缓冲区、文件描述符上限、连接队列

**数据库基础设施**
- 连接池大小调优
- 主从复制延迟监控
- 定期归档历史数据，保持热数据表体量可控

### 6. 度量与监控

**关键指标**
- 响应时间：P50 / P95 / P99
- 吞吐量：QPS / TPS
- 错误率：5xx 比例
- 资源利用率：CPU、内存、磁盘 IO、网络带宽

**工具链**
- APM：SkyWalking、Jaeger（链路追踪）
- 日志：ELK / Loki（结构化日志 + 聚合分析）
- 指标：Prometheus + Grafana
- 压测：JMeter、wrk、k6（上线前验证容量）

## 易错点

- 缓存和数据库一致性问题处理不当
- 连接池配置不合理导致连接耗尽
- 异步化后没有做好补偿和重试机制
- 过度微服务化导致网络调用链过长
