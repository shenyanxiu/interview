# Node.js 模块化机制

## 问题

Node.js 的模块化机制是怎样的？CommonJS 和 ES Modules 有什么区别？

## 主题

Node.js / 模块化

## 难度

中级

## 参考答案

Node.js 有两套模块系统：**CommonJS (CJS)** 和 **ES Modules (ESM)**。

### CommonJS（传统方式）

Node.js 最初的模块系统，同步加载。

```javascript
// 导出
module.exports = { add, subtract };
// 或
exports.add = function(a, b) { return a + b; };

// 导入
const math = require('./math');
const { add } = require('./math');
```

特点：
- `require()` 是同步的，运行时加载
- `module.exports` 是真正的导出对象，`exports` 只是它的引用
- 模块有缓存，第一次 `require` 后会缓存，后续返回同一个对象
- 循环依赖时返回未完成的副本（部分导出）

常见坑：

```javascript
// ❌ 断开 exports 和 module.exports 的引用
exports = { foo: 'bar' };

// ✅ 正确写法
module.exports = { foo: 'bar' };
```

### ES Modules（现代方式）

```javascript
// 导出
export function add(a, b) { return a + b; }
export default class Calculator {}

// 导入
import { add } from './math.mjs';
import Calculator from './math.mjs';
import * as math from './math.mjs';
```

启用方式（三选一）：
1. 文件后缀用 `.mjs`
2. `package.json` 中设置 `"type": "module"`
3. 命令行 `--input-type=module`

特点：
- 静态分析，编译时确定依赖关系（支持 tree-shaking）
- 异步加载
- 导出的是绑定（live binding），不是值的拷贝
- 严格模式，顶层 `this` 是 `undefined`

### CJS vs ESM 核心区别

| | CommonJS | ES Modules |
|---|---|---|
| 语法 | `require` / `module.exports` | `import` / `export` |
| 加载时机 | 运行时（动态） | 编译时（静态） |
| 导出本质 | 值的拷贝 | 值的引用（live binding） |
| 顶层 await | ❌ | ✅ |
| `__dirname` / `__filename` | ✅ 直接可用 | ❌ 需要自己构造 |
| 循环依赖 | 返回未完成的导出对象 | 拿到引用但值可能未初始化 |

### ESM 中获取 __dirname

```javascript
import { fileURLToPath } from 'url';
import { dirname } from 'path';

const __filename = fileURLToPath(import.meta.url);
const __dirname = dirname(__filename);
```

### CommonJS 常见模式

```javascript
// 单例模式（模块缓存天然支持）
const connection = createConnection();
module.exports = connection;

// 工厂模式
module.exports = function(prefix) {
  return {
    log(msg) { console.log(`[${prefix}] ${msg}`); }
  };
};
const logger = require('./logger')('App');

// 条件导入（动态，CJS 独有优势）
const db = process.env.NODE_ENV === 'test'
  ? require('./mock-db')
  : require('./real-db');
```

### 模块包装

Node.js 实际上会把每个 CJS 文件包在一个函数里执行：

```javascript
(function(exports, require, module, __filename, __dirname) {
  // 你的代码在这里
});
```

所以 `exports`、`require`、`module`、`__filename`、`__dirname` 不是全局变量，是这个包装函数的参数。

### 模块解析规则

`require('xxx')` 的查找顺序：

1. **核心模块** — `fs`, `path` 等内置模块，优先级最高
2. **路径模块** — 以 `./`、`../`、`/` 开头，按路径查找
3. **第三方模块** — 从当前目录的 `node_modules` 逐级向上查找，直到根目录

文件查找顺序：`xxx` → `xxx.js` → `xxx.json` → `xxx.node` → `xxx/index.js`

### package.json 中的模块相关字段

```json
{
  "main": "./lib/index.cjs",
  "module": "./lib/index.mjs",
  "type": "module",
  "exports": {
    ".": {
      "import": "./lib/index.mjs",
      "require": "./lib/index.cjs"
    },
    "./utils": "./lib/utils.mjs"
  }
}
```

# .mjs / .cjs / .js 的使用场景

## 问题

什么情况下用 `.mjs`，什么情况下用 `.cjs`，什么情况下用 `.js`？

## 主题

Node.js / 模块化

## 难度

中级

## 参考答案

核心规则：**后缀决定单个文件的模块类型，`package.json` 的 `"type"` 字段决定 `.js` 文件的默认解析方式。**

### `.js` — 取决于 package.json

- `"type": "module"` → `.js` 按 ESM 解析
- `"type": "commonjs"` 或没写 → `.js` 按 CJS 解析

这是最常见的用法。大多数项目统一用 `.js`，靠 `"type"` 字段控制。

### `.mjs` — 强制 ESM

不管 `package.json` 怎么配，`.mjs` 永远是 ES Module。

使用场景：
- 项目整体是 CJS，但个别文件需要用 ESM 语法
- 发布 npm 包时同时提供 CJS 和 ESM 两份产物，ESM 那份用 `.mjs`

### `.cjs` — 强制 CJS

不管 `package.json` 怎么配，`.cjs` 永远是 CommonJS。

使用场景：
- 项目整体是 ESM（设了 `"type": "module"`），但某些配置文件必须用 CJS（如 `eslint.config.cjs`、`jest.config.cjs`）
- 发布 npm 包时的 CJS 产物

### 实际选择建议

| 场景 | 推荐做法 |
|---|---|
| 新项目，纯 ESM | `"type": "module"` + 全用 `.js`，个别需要 CJS 的配置文件用 `.cjs` |
| 老项目，纯 CJS | 不设 type + 全用 `.js`，想尝试 ESM 的文件用 `.mjs` |
| npm 包双格式发布 | 产物分 `.mjs` 和 `.cjs`，通过 `exports` 字段映射 |
| 不确定 | 用显式后缀 `.mjs` / `.cjs`，最不容易出错 |

简单总结：**`.js` 是"听 package.json 的"，`.mjs` 和 `.cjs` 是"我说了算"。**

- 忘记某些工具（如 Jest、ESLint 旧版）不支持 ESM 配置文件，需要用 `.cjs`

# CJS 和 ESM 能否互相引用

## 问题

CommonJS 和 ES Modules 能互相引用吗？有什么限制？

## 主题

Node.js / 模块化

## 难度

中级

## 参考答案

能，但不对等，有方向性限制。

### ESM 引入 CJS ✅ 简单

```javascript
import pkg from './cjs-file.cjs';          // module.exports 作为默认导出
import { method } from './cjs-file.cjs';   // Node 尝试静态分析具名导出（不保证）
```

- CJS 的 `module.exports` 整体变成 ESM 的 `default` 导出
- Node.js 会尝试解析出具名导出，但不保证都能识别
- 如果具名导入失败，用默认导入再解构：`const { method } = pkg`

### CJS 引入 ESM ⚠️ 受限

```javascript
// ❌ 不行，require 是同步的，ESM 是异步的
const esm = require('./esm-file.mjs');

// ✅ 只能用动态 import()
async function main() {
  const { default: foo, bar } = await import('./esm-file.mjs');
}
main();
```

- `require()` 无法加载 ESM，会直接报错
- 只能用 `import()`（返回 Promise），必须在异步上下文中使用

### 对比总结

| 方向 | 能否引用 | 方式 | 痛点 |
|---|---|---|---|
| ESM → CJS | ✅ 直接 | `import` | 具名导出识别不一定准 |
| CJS → ESM | ⚠️ 受限 | `await import()` | 必须异步，不能用 require |

### 根本原因

**ESM 是异步加载的，CJS 是同步的。**

- `import` 语句本身支持异步解析，加载 CJS（同步）没问题
- `require()` 是同步调用，没法等一个异步的 ESM 模块加载完成

这也是为什么很多 npm 包还在提供 CJS 版本——如果只发 ESM，所有 CJS 消费者都得改成异步写法。

## 易错点

- 忽略 `import()` 返回的是 Promise，忘记 await

- 各大框架（Next.js、Nuxt）处理 CJS/ESM 互操作的策略
