# V8 执行流程

## 问题

V8 引擎的执行流程是怎样的？

## 难度

中级

## 参考答案

V8 是 Chrome 和 Node.js 使用的 JavaScript 引擎，采用 **Ignition（解释器）+ TurboFan（优化编译器）** 的流水线架构。

### 整体流程

```
源代码 → Scanner（词法分析）→ Tokens
       → Parser（语法分析）→ AST
       → Ignition（解释器）→ 字节码 → 逐条执行
                                │
                                │ 热点代码
                                ▼
                           TurboFan（优化编译器）→ 机器码
                                │
                                │ 类型假设失败
                                ▼
                            去优化（Deopt）→ 回退到字节码执行
```

### 1. 源码 → AST（解析阶段）

- **词法分析（Scanner）**：将源代码拆分为 Token（关键字、标识符、运算符等）
- **语法分析（Parser）**：将 Token 转换为 AST（抽象语法树）
  - **Pre-Parser（预解析）**：快速扫描，跳过函数体内部（惰性解析），减少启动时间
  - **Full Parser（全量解析）**：函数真正被调用时才完整解析

### 2. AST → 字节码（Ignition 解释器）

AST 交给 Ignition 编译为字节码，逐条解释执行，同时收集**类型反馈信息**（Type Feedback）。

### 3. 热点代码 → 机器码（TurboFan 优化编译器）

当某段代码被反复执行（热点代码），TurboFan 将其编译为高度优化的机器码。优化手段包括：

- 内联缓存（Inline Cache）
- 内联展开（Inlining）
- 逃逸分析
- 类型特化

### 4. 去优化（Deoptimization）

运行时类型发生变化时，优化假设被打破，回退到 Ignition 解释执行字节码。

```js
function add(a, b) {
  return a + b;
}

// 前 10000 次都是 number，TurboFan 优化为整数加法
for (let i = 0; i < 10000; i++) add(i, i);

// 突然传入 string，触发去优化
add("hello", "world");
```

- 什么是内联缓存（Inline Cache）？单态、多态、超态分别是什么？
- 如何写出对 V8 友好的代码？
- 什么是 Hidden Class？它如何加速属性访问？

## 易错点

- 误以为 V8 直接将 JS 编译为机器码（实际先生成字节码）
- 忽略去优化的存在，认为代码一旦被优化就永远以机器码运行
- 混淆 AST 和字节码的概念

# 字节码与机器码

## 问题

字节码是什么？机器码是什么？它们有什么区别？

## 难度

初级

## 参考答案

### 机器码（Machine Code）

CPU 能直接执行的二进制指令，与具体 CPU 架构绑定（x86、ARM 等）。

特点：
- 执行最快，CPU 直接识别
- 平台相关，不可跨架构
- 体积大，不可读

### 字节码（Bytecode）

介于源代码和机器码之间的**中间表示**，给虚拟机看的"伪指令"，不能被 CPU 直接执行。

```
// V8 Ignition 字节码示例
LdaNamedProperty a0, [0]   // 加载参数 a
Add a1                      // 加上参数 b
Return                      // 返回结果
```

特点：
- 平台无关，可跨平台运行
- 体积小，生成速度快
- 需要解释器逐条翻译执行

### 对比

| | 字节码 | 机器码 |
|---|---|---|
| 执行者 | 虚拟机 / 解释器 | CPU 硬件 |
| 执行速度 | 较慢（需翻译） | 最快（直接执行） |
| 跨平台 | ✅ 是 | ❌ 否 |
| 体积 | 小 | 大 |
| 生成速度 | 快 | 慢（优化编译耗时） |

### 在 V8 中的关系

```
JS 源码 → 字节码（Ignition 快速生成，立即执行）
              │ 热点代码
              ▼
         机器码（TurboFan 深度优化，执行极快）
```

这就是 JIT 策略：冷代码跑字节码省编译时间，热代码编译成机器码获得极致性能。

### 其他语言中的例子

- **Java**：`.java` → `.class` 字节码 → JVM 执行
- **Python**：`.py` → `.pyc` 字节码 → CPython 执行
- **V8**：JS 源码 → Ignition 字节码 → TurboFan 机器码

## 追问链

- 为什么 V8 不直接编译成机器码？
- JIT 和 AOT 编译的区别是什么？
- WebAssembly 的字节码和 V8 字节码有什么不同？

## 易错点

- 误以为字节码就是机器码（字节码不能被 CPU 直接执行）
- 误以为所有字节码最终都会变成机器码（只有热点代码才会）
- 混淆"编译型语言"和"解释型语言"的边界（现代引擎大多是混合模式）

## 知识延伸

- WebAssembly（Wasm）作为浏览器中的另一种字节码格式
- V8 的 `--print-bytecode` 标志可以查看生成的字节码
- 分层编译（Tiered Compilation）在 JVM 和 V8 中的应用
