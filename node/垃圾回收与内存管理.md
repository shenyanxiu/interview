# Node.js 垃圾回收与内存管理

## 问题 1：Node.js 的垃圾回收与内存管理机制是什么样的？

### 主题
Node.js / V8 内存管理

### 难度
中级

### 参考答案

Node.js 使用 V8 引擎的内存管理机制，采用分代回收策略。

**内存结构：**
- **新生代（New Space）**：存放短生命周期对象，空间小（通常 1-8MB），GC 频繁
- **老生代（Old Space）**：存放长生命周期对象，空间大（默认约 1.4GB/64位），GC 较少
- **大对象空间（Large Object Space）**：超过一定大小的对象直接分配在这里
- **代码空间（Code Space）**：存放 JIT 编译后的代码
- **Map 空间**：存放对象的隐藏类（Hidden Class）

**垃圾回收算法：**
- 新生代使用 Scavenge（Cheney 算法），将空间分为 From/To 两个半区，存活对象复制后交换
- 经历两次 Scavenge 仍存活的对象晋升到老生代
- 老生代使用 Mark-Sweep（标记清除）+ Mark-Compact（标记整理）
- V8 使用增量标记、并发标记来减少主线程停顿

**常见内存泄漏场景：**
- 全局变量未释放
- 闭包引用了不再需要的大对象
- 未清除的定时器和事件监听器
- 缓存无限增长
- 被遗忘的 Stream 或 Socket 连接

**实用手段：**
```js
process.memoryUsage() // { rss, heapTotal, heapUsed, external, arrayBuffers }
// node --max-old-space-size=4096 app.js
// --inspect + Chrome DevTools Memory 面板
```

### 追问链
- V8 的增量标记是如何实现的？（三色标记法）
- 新生代对象晋升到老生代的条件是什么？
- 如何排查线上 Node.js 内存泄漏？

### 易错点
- 误以为 Node.js 使用引用计数（实际使用标记清除）
- 混淆 rss 和 heapUsed 的含义
- 认为手动设置 null 就能立即释放内存（只是解除引用，回收时机由 GC 决定）

### 知识延伸
- WeakRef 和 FinalizationRegistry（ES2021）
- Buffer 和 ArrayBuffer 的内存分配在 V8 堆外
- Worker Threads 各自拥有独立的 V8 实例和堆

## 问题 2：引用计数和标记清除的区别是什么？

### 主题
Node.js / V8 内存管理

### 难度
中级

### 参考答案

**引用计数（Reference Counting）：**
- 每个对象维护一个计数器，记录有多少引用指向它
- 引用增加计数+1，引用移除计数-1，计数归零时立即回收
- 优点：回收即时，不需要暂停程序
- 致命缺陷：无法处理循环引用

```js
function problem() {
  let a = {};
  let b = {};
  a.ref = b; // b 的计数 = 1
  b.ref = a; // a 的计数 = 1
  // 函数结束后互相引用，计数永远不为 0，无法回收
}
```

**标记清除（Mark-Sweep）：**
- 从根对象（GC Roots）出发，遍历所有可达对象并标记
- 清除所有未被标记的对象
- 天然解决循环引用（从根不可达的对象照样被回收）
- 缺点：需要 Stop-The-World，清除后产生内存碎片

**对比：**

| | 引用计数 | 标记清除 |
|---|---|---|
| 回收时机 | 即时 | GC 触发时 |
| 循环引用 | ❌ 无法处理 | ✅ 自然解决 |
| 停顿 | 无 | 有（STW） |
| 内存碎片 | 无 | 有（需配合 Mark-Compact） |
| V8 是否使用 | 否 | 是（主算法） |

**V8 的优化：**
- 增量标记：把标记拆成小步，穿插在主线程执行中
- 并发标记：标记工作放到辅助线程
- 并行清除：多线程同时清除
- 懒清除：按需清除，不一次性清除所有垃圾

### 追问链
- 三色标记法是什么？如何配合增量标记？
- 写屏障（Write Barrier）的作用是什么？
- Python 是如何解决引用计数的循环引用问题的？

### 易错点
- 误以为现代 JS 引擎仍使用引用计数
- 混淆"不可达"和"没有引用"（循环引用的对象有引用但不可达）
- 认为标记清除会导致明显卡顿（现代 V8 优化后停顿通常只有几毫秒）

### 知识延伸
- 早期 IE 的 DOM 对象使用引用计数，导致大量内存泄漏
- Python 使用引用计数 + 循环检测器的混合方案
- Rust 的所有权系统在编译期解决内存管理，无需 GC
