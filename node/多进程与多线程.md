# 多进程与多线程

## Q1: child_process / cluster / worker_threads 的区别

### 问题

Node.js 中 child_process、cluster、worker_threads 三者有什么区别？分别在什么场景使用？

### 参考答案

| 维度 | child_process | cluster | worker_threads |
|------|--------------|---------|----------------|
| 本质 | 创建子进程 | 基于 child_process 的多进程封装 | 创建子线程 |
| 隔离性 | 完全独立的内存空间 | 完全独立的内存空间 | 共享进程内存（可用 SharedArrayBuffer） |
| 通信方式 | IPC（send / message 事件） | IPC（同 child_process） | postMessage 或 SharedArrayBuffer |
| 开销 | 大（新进程） | 大（新进程） | 较小（新线程） |
| 适用场景 | 执行外部命令、运行独立脚本 | Web 服务器多核负载均衡 | CPU 密集型计算 |

**选择指南：**
- 调用 Python/Shell/ffmpeg 等外部程序 → `child_process.spawn / exec`
- 把耗时任务丢给另一个 Node.js 脚本 → `child_process.fork`
- Web 服务要利用多核 CPU 提高并发 → `cluster`
- Node.js 内做 CPU 密集计算（加密、压缩、图像处理） → `worker_threads`

### 追问链

- cluster 底层是怎么实现端口共享的？
- worker_threads 的 SharedArrayBuffer 怎么用？需要注意什么？
- PM2 和 cluster 是什么关系？

### 易错点

- 误认为 cluster 是多线程——它是多进程，每个 worker 是独立进程
- 误认为 worker_threads 和 Web Worker 完全一样——worker_threads 有完整 Node API

### 知识延伸

- PM2 是 cluster 模式的生产级封装，提供进程守护、日志管理、负载均衡
- Deno/Bun 对多线程的支持方式

## Q2: worker_threads 和 Web Worker 的对比

### 问题

Node.js 的 worker_threads 和浏览器的 Web Worker 是不是一样的？

### 参考答案

设计理念相同（主线程之外开线程跑 JS，通过消息通信），但运行环境和能力有差异：

| 维度 | worker_threads (Node.js) | Web Worker (浏览器) |
|------|--------------------------|---------------------|
| 运行环境 | Node.js | 浏览器 |
| 共享内存 | ✅ SharedArrayBuffer + Atomics | ✅ 需 COOP/COEP 头 |
| 访问能力 | 完整 Node API（fs、net、crypto 等） | 受限，无 DOM，有 fetch、IndexedDB |
| 模块系统 | CommonJS / ESM 都支持 | 仅 ESM |
| 创建方式 | `new Worker(file, { workerData })` | `new Worker(url)` |

**相似之处：**
- 都不能访问主线程的变量，通过消息传递通信
- 都用 postMessage / onmessage 通信
- 都支持 Transferable Objects（零拷贝转移 ArrayBuffer）
- 都适合 CPU 密集任务

**最大区别：** worker_threads 能用完整 Node.js 能力，Web Worker 运行在浏览器沙箱中不能碰 DOM。

**浏览器端 Worker 变体：**
- Dedicated Worker：一对一绑定页面
- Shared Worker：多个同源页面共享
- Service Worker：独立于页面生命周期，拦截网络请求，实现离线缓存

### 追问链

- Service Worker 的生命周期是怎样的？
- SharedArrayBuffer 为什么需要 COOP/COEP 头？（Spectre 漏洞）
- Transferable 和 Structured Clone 的区别？

### 易错点

- 认为 Web Worker 可以操作 DOM——不行，只有主线程能操作 DOM
- 认为 SharedArrayBuffer 在浏览器中可以随意使用——需要设置安全头

## Q3: child_process 和 cluster 的关系

### 问题

child_process 和 cluster 到底是什么关系？为什么有了 child_process 还需要 cluster？

### 参考答案

**关系：** cluster 是 `child_process.fork()` + 端口共享 + 请求分发的上层封装。

```
child_process（底层能力）
    │
    ├── exec / spawn → 执行任意外部命令
    │
    └── fork → 创建 Node.js 子进程
         │
         └── cluster（上层封装）
              → 自动 fork 多份相同服务
              → 自动共享端口
              → 自动分发请求（round-robin）
```

**为什么需要 cluster：**

没有 cluster 时，如果你想让 4 个 Node.js 进程处理 HTTP 请求，你需要手动：
1. fork 4 个子进程
2. 解决端口冲突问题
3. 自己实现请求分发逻辑

cluster 把这些全做了——多个工作进程 listen 同一个端口不冲突，主进程自动做 round-robin 分发。

**类比：** child_process 是"你有能力开新店"，cluster 是"把同一家店复制多份，共用一个大门，自动分流客人"。

### 追问链

- cluster 的 round-robin 是在哪一层实现的？
- Windows 和 Linux 上 cluster 的分发策略有什么不同？
- 如何实现 cluster 工作进程的优雅重启（zero-downtime restart）？

### 易错点

- 认为 cluster 可以执行外部命令——它只能 fork 当前 Node.js 脚本的副本
- 认为 cluster 的工作进程之间可以共享内存——它们是独立进程，内存隔离

## Q4: spawn 和 exec 的区别

### 问题

child_process 中 spawn 和 exec 有什么区别？什么时候用哪个？

### 参考答案

**一句话区分：** exec 把结果攒完一次性给你，spawn 边产出边给你（流式）。

| 维度 | exec | spawn |
|------|------|-------|
| 输出方式 | 缓冲到内存，结束后一次性回调 | 流式（stream），实时获取 |
| 缓冲区限制 | 默认 ~1MB，超出报错 | 无限制 |
| Shell | 默认通过 shell 执行（支持管道、通配符） | 默认不经过 shell（更安全） |
| 返回值 | 回调里拿到完整 stdout/stderr 字符串 | 返回 ChildProcess 对象，监听 data 事件 |
| 适合场景 | 输出少、一次性拿结果 | 输出大、需要实时处理 |

**exec 的优势——支持 shell 语法：**
```js
exec('ps aux | grep node | wc -l', (err, stdout) => {
  console.log('Node 进程数:', stdout.trim());
});
```

**spawn 的优势——流式处理大输出：**
```js
const child = spawn('find', ['/', '-name', '*.log']);
child.stdout.on('data', (chunk) => {
  console.log(chunk.toString()); // 实时收到
});
```

**选择：**
- 输出小 + 想用 shell 语法 → exec
- 输出大 / 需要实时处理 / 长时间运行 → spawn

另外还有 `execFile`：和 exec 类似（缓冲输出），但不经过 shell，直接执行可执行文件，更安全、更快。

### 追问链

- exec 的 maxBuffer 超出后怎么处理？
- spawn 的 `{ shell: true }` 选项有什么安全风险？
- fork 和 spawn 的区别是什么？

### 易错点

- 用 exec 处理大文件输出导致 maxBuffer exceeded 错误
- 在 exec 中拼接用户输入导致命令注入漏洞（因为它走 shell）
- 误认为 spawn 不能用 shell 语法——可以通过 `{ shell: true }` 开启，但失去了安全优势

### 知识延伸

- `execFile`：不走 shell 的 exec，更安全
- `fork`：专门创建 Node.js 子进程的 spawn，自带 IPC 通道
- 四者关系：spawn 是最底层，exec/execFile/fork 都是基于 spawn 的封装
