# Node.js 事件循环（Event Loop）

## 问题

Node.js 的事件循环是如何工作的？请描述其各个阶段及执行顺序。

## 主题

Node.js / 异步编程

## 难度

中级

## 参考答案

Node.js 的事件循环是其非阻塞 I/O 模型的核心机制，让单线程的 Node.js 能够高效处理大量并发操作。

### 事件循环的六个阶段

每次迭代（一个 tick）按以下顺序执行：

```
   ┌───────────────────────────┐
┌─>│        timers              │  setTimeout / setInterval 回调
│  └───────────┬───────────────┘
│  ┌───────────┴───────────────┐
│  │     pending callbacks      │  系统级回调（如 TCP 错误）
│  └───────────┬───────────────┘
│  ┌───────────┴───────────────┐
│  │       idle, prepare        │  内部使用
│  └───────────┬───────────────┘
│  ┌───────────┴───────────────┐
│  │         poll               │  获取新的 I/O 事件，执行 I/O 回调
│  └───────────┬───────────────┘
│  ┌───────────┴───────────────┐
│  │         check              │  setImmediate 回调
│  └───────────┬───────────────┘
│  ┌───────────┴───────────────┐
│  │     close callbacks        │  如 socket.on('close')
└──┴───────────────────────────┘
```

1. **timers** — 执行到期的 `setTimeout` 和 `setInterval` 回调。定时器的延迟是最小延迟，不是精确延迟。
2. **pending callbacks** — 执行推迟到下一次循环的系统操作回调（如某些 TCP 错误）。
3. **idle, prepare** — 仅供内部使用。
4. **poll** — 最重要的阶段。检索新的 I/O 事件，执行 I/O 相关回调。如果队列为空，会在此阻塞等待，直到有新事件或定时器到期。
5. **check** — 执行 `setImmediate()` 回调。poll 阶段完成后立即进入此阶段。
6. **close callbacks** — 执行关闭事件的回调，如 `socket.on('close', ...)`。

### 微任务队列（优先级最高）

在每个阶段切换之间，Node.js 会清空两个微任务队列：

- `process.nextTick()` — 优先级最高，在当前操作完成后立即执行
- `Promise.then()` / `queueMicrotask()` — 次于 nextTick，但仍优先于下一个阶段

```javascript
setTimeout(() => console.log("timeout"), 0);
setImmediate(() => console.log("immediate"));
process.nextTick(() => console.log("nextTick"));
Promise.resolve().then(() => console.log("promise"));

// 输出顺序：
// nextTick
// promise
// timeout (或 immediate，取决于执行时机)
// immediate (或 timeout)
```

### setTimeout vs setImmediate

- 在主模块中，两者顺序不确定（取决于进程性能）
- 在 I/O 回调内部，`setImmediate` 总是先于 `setTimeout`

```javascript
const fs = require("fs");

fs.readFile(__filename, () => {
  setTimeout(() => console.log("timeout"), 0);
  setImmediate(() => console.log("immediate"));
});
// 始终输出：immediate -> timeout
```

# setImmediate 是什么

## 问题

`setImmediate` 是什么？它和 `setTimeout(fn, 0)` 有什么区别？

## 主题

Node.js / 异步编程

## 难度

中级

## 参考答案

`setImmediate(callback)` 是 Node.js 提供的定时器函数，用于将回调安排在当前 poll 阶段完成后、事件循环进入 **check 阶段**时执行。

### 核心特点

- 不接受延迟参数，语义是"当前 I/O 事件处理完后尽快执行"
- 在事件循环的 check 阶段执行
- 每次事件循环迭代只执行一批，不会像 `process.nextTick` 那样阻塞循环
- 返回 `Immediate` 对象，可用 `clearImmediate()` 取消
- Node.js 特有 API，浏览器中不是标准（IE 曾实现过，已废弃）

### 与 setTimeout(fn, 0) 的对比

| | `setImmediate` | `setTimeout(fn, 0)` |
|---|---|---|
| 执行阶段 | check | timers |
| 在 I/O 回调中 | 始终先执行 | 始终后执行 |
| 在主模块中 | 顺序不确定 | 顺序不确定 |

### 典型使用场景

在 I/O 回调中，需要确保某段逻辑在当前 I/O 处理完之后、下一轮 timers 之前执行：

```javascript
const fs = require('fs');

fs.readFile('file.txt', (err, data) => {
  // 处理数据...
  setImmediate(() => {
    console.log('在 check 阶段执行，优先于下一轮 timers');
  });
});
```

## 追问链

- setImmediate 和 requestAnimationFrame 有什么关系？
- 浏览器中有没有等价的 API？

## 易错点

- 误以为 setImmediate 是"立即执行"，实际上要等到 check 阶段
- 在主模块中假设 setImmediate 一定比 setTimeout(fn, 0) 先执行（顺序不确定）

# 事件循环每个阶段之间都执行 nextTick 和 then 吗

## 问题

事件循环的六个阶段，每个阶段之间都会执行 `process.nextTick` 和 `Promise.then` 吗？

## 主题

Node.js / 异步编程

## 难度

高级

## 参考答案

是的。在 Node.js 11+ 中，**每个阶段切换之间**以及**每个回调执行完毕后**都会清空 nextTick 队列和微任务队列。

### 执行模型（Node.js 11+）

```text
timers 阶段:
  执行 timer 回调 1 → 清空 nextTick → 清空 Promise
  执行 timer 回调 2 → 清空 nextTick → 清空 Promise
  ...
─── 阶段切换 → 清空 nextTick → 清空 Promise ───
pending callbacks 阶段:
  执行回调 → 清空 nextTick → 清空 Promise
  ...
─── 阶段切换 → 清空 nextTick → 清空 Promise ───
poll 阶段:
  ...
─── 阶段切换 → 清空 nextTick → 清空 Promise ───
check 阶段:
  ...
（以此类推）
```

### Node.js 11 之前的行为（旧版）

在 Node.js 10 及更早版本中，微任务是在**整个阶段的所有回调执行完**之后才统一清空的：

```javascript
// Node 10 vs Node 11+ 行为差异
setTimeout(() => {
  console.log('timer1');
  Promise.resolve().then(() => console.log('promise1'));
}, 0);
setTimeout(() => {
  console.log('timer2');
  Promise.resolve().then(() => console.log('promise2'));
}, 0);

// Node 10: timer1 → timer2 → promise1 → promise2
// Node 11+: timer1 → promise1 → timer2 → promise2
```

Node.js 11+ 的行为与浏览器保持一致——每个宏任务执行完就立即清空微任务队列。

## 追问链

- Node.js 11 为什么要改变这个行为？
- 这个变更会导致哪些旧代码出现兼容性问题？
- 浏览器的事件循环在这方面是怎样的？

## 易错点

- 用 Node.js 10 的心智模型理解 11+ 的行为，导致对执行顺序判断错误
- 误以为微任务只在阶段切换时执行，实际上每个回调之后都会清空

## 知识延伸

- Node.js 11 changelog 中关于此变更的说明
- 浏览器事件循环规范中微任务检查点（microtask checkpoint）的定义

# nextTick 为什么比 then 先执行

## 问题

`process.nextTick` 为什么比 `Promise.then` 先执行？

## 主题

Node.js / 异步编程

## 难度

高级

## 参考答案

这是 Node.js 的**设计决策**，不是 JavaScript 语言规范要求的。

### 根本原因：两个独立的队列

Node.js 内部维护了两个不同的队列：

- **nextTick 队列** — Node.js 自己实现的，属于 Node 内部机制
- **微任务队列（microtask queue）** — V8 引擎管理的，Promise.then / queueMicrotask 进入这里

在每次清空微任务的时机，Node.js 的处理逻辑是：

```c
// 伪代码，简化自 Node.js 源码
while (nextTickQueue.length || microtaskQueue.length) {
  // 先清空 nextTick
  while (nextTickQueue.length) {
    runNextTick();
  }
  // 再清空微任务
  while (microtaskQueue.length) {
    runMicrotask();
  }
}
```

nextTick 先执行纯粹是因为代码里**先检查它**。

### 历史原因

`process.nextTick` 比 Promise 出现得早得多（Node.js 0.1 就有了，Promise 是 ES2015 才进入标准）。它最初的语义就是"在当前操作完成后、任何 I/O 之前立即执行"，是 Node.js 给开发者提供的最高优先级异步插队机制。后来 Promise 进入 Node.js 后，为了保持向后兼容，nextTick 的优先级被保留在 Promise 之上。

### 验证示例

```javascript
process.nextTick(() => {
  console.log('nextTick 1');
  Promise.resolve().then(() => console.log('promise inside nextTick'));
  process.nextTick(() => console.log('nextTick 2'));
});

Promise.resolve().then(() => console.log('promise 1'));

// 输出：
// nextTick 1
// nextTick 2
// promise inside nextTick
// promise 1
```

注意 `nextTick 2` 在 `promise inside nextTick` 之前——因为 nextTick 队列要完全清空后才轮到微任务队列。
