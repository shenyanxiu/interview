# 工厂模式与 new 的区别

## 问题

工厂模式和 new 的区别是什么？

## 主题

设计模式 - 创建型

## 难度

初级

## 参考答案

核心区别：**谁来决定创建什么对象，以及调用方是否需要知道具体类。**

### 直接 new

```js
// 调用方必须知道具体类
const dog = new Dog();
const cat = new Cat();
```

调用方与具体类强耦合——必须知道类名、构造参数，如果类变了，所有 `new` 的地方都要改。

### 工厂模式

```js
function createAnimal(type) {
  switch (type) {
    case 'dog': return new Dog();
    case 'cat': return new Cat();
    default: throw new Error(`Unknown type: ${type}`);
  }
}

// 调用方只需要知道类型标识
const animal = createAnimal('dog');
```

调用方不关心具体类是什么，创建逻辑集中在工厂里。

### 关键差异对比

| | 直接 new | 工厂模式 |
|---|---|---|
| 耦合度 | 调用方依赖具体类 | 调用方只依赖工厂接口 |
| 灵活性 | 换实现要改所有调用处 | 只改工厂内部 |
| 封装创建逻辑 | 无，散落各处 | 集中管理 |
| 适用场景 | 类型确定、简单创建 | 类型不确定、创建逻辑复杂 |

### 什么时候用工厂

1. **创建逻辑复杂** — 需要根据配置、环境、参数决定实例化哪个类
2. **解耦** — 不想让业务代码直接依赖具体实现类
3. **统一入口** — 方便加缓存、日志、权限校验等横切逻辑

```js
// 实际场景：根据环境创建不同的日志实例
function createLogger(env) {
  if (env === 'production') {
    return new CloudLogger({ level: 'warn' });
  }
  return new ConsoleLogger({ level: 'debug' });
}
```

### 什么时候直接 new 就够了

类型明确、不会变化、创建逻辑简单——没必要为了模式而模式。

```js
const map = new Map();
const set = new Set();
```

## 易错点

- 认为工厂模式就是不用 new（工厂内部还是会 new，只是把 new 的决策封装了）
- 所有场景都用工厂（简单场景直接 new 更清晰，过度设计反而增加复杂度）

## 知识延伸

- 工厂模式本质是对 new 的一层抽象，把"创建什么"的决策从调用方转移到工厂内部
- 当创建逻辑有变化的可能性时，工厂模式的价值就体现出来了
- 与依赖注入（DI）容器的关系
