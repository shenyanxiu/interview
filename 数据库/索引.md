# 索引

## 数据库索引是什么

### 问题

数据库索引是什么？

### 参考答案

数据库索引是一种**用空间换时间的数据结构**，作用类似书的目录——不用逐页翻找，直接定位到目标位置。

索引是独立于表数据之外的一份有序数据结构，存储了索引列的值和对应行的物理地址（或主键），让数据库能快速定位数据而不必全表扫描。

**主流实现**

| 类型 | 结构 | 适用场景 |
|------|------|----------|
| B+树索引 | 多路平衡树，叶子节点有序链表 | 范围查询、排序、最常用 |
| 哈希索引 | 哈希表 | 等值查询极快，不支持范围 |
| 全文索引 | 倒排索引 | 文本搜索 |
| R树索引 | 空间索引 | 地理/空间数据 |

**索引分类**

- 主键索引（聚簇索引）：叶子节点直接存整行数据，表数据按主键物理排序
- 二级索引（非聚簇索引）：叶子节点存主键值，查到后需要回表
- 覆盖索引：查询字段全在索引中，无需回表
- 联合索引：多列组合，遵循最左前缀匹配

**代价**

- 占用额外存储空间
- INSERT/UPDATE/DELETE 时需要同步维护索引，写入变慢
- 索引过多会导致优化器选择困难

### 易错点

- 索引不是越多越好，写多读少的表要控制索引数量
- LIKE '%keyword' 会导致索引失效
- 对索引列做函数运算会导致索引失效

### 知识延伸

- B+树的结构与查找过程
- 聚簇索引 vs 非聚簇索引
- 索引下推（Index Condition Pushdown）

## 索引的大致原理（B+树）

### 问题

索引的大致原理是什么？

### 参考答案

以最常用的 B+树索引为例：

**问题背景**：表数据存在磁盘上，没有索引时查找只能全表扫描，数据量大时极慢。

**B+树结构**

```
         [30, 60]              ← 根节点（第1次IO）
        /    |    \
   [10,20] [35,50] [70,80]    ← 中间节点（第2次IO）
    / | \    / | \    / | \
  叶子 叶子 叶子 叶子 叶子 叶子  ← 叶子节点（第3次IO）
  ←→  ←→  ←→  ←→  ←→         ← 叶子间双向链表
```

**查找过程**（以 `age = 35` 为例）：
1. 读根节点：35 在 30~60 之间，走中间指针
2. 读中间节点：35 等于第一个键，走对应指针
3. 读叶子节点：找到 35 对应的数据行地址

3 次磁盘 IO 就定位到了。

**为什么树很矮**

一个节点大小通常等于一个磁盘页（16KB），假设每个键+指针占 12 字节，一个节点能放约 1000 个指针：
- 2 层：1000² = 100 万行
- 3 层：1000³ = 10 亿行

千万级数据，树高也就 3~4 层，即 3~4 次 IO。

**聚簇 vs 非聚簇**
- 聚簇索引（主键）：叶子节点直接存完整行数据，找到就拿到了
- 非聚簇索引（二级索引）：叶子节点存的是主键值，找到后还要回表

**范围查询为什么也快**

叶子节点之间用链表串起来，找到起点后顺着链表往后扫就行。比如 `WHERE age BETWEEN 20 AND 40`，定位到 20 后沿链表扫到 40 即可。

**一句话总结**：索引把无序的磁盘数据用有序的树结构做"目录"，将随机 IO 转化为少量顺序查找，从 O(n) 降到 O(log n)。

### 易错点

- B+树不是二叉树，是多路平衡树（扇出很大）
- 非叶子节点不存数据，只存键值和指针
- 回表是二级索引的额外开销，覆盖索引可以避免

### 知识延伸

- B树 vs B+树的区别
- 页分裂与页合并
- 自适应哈希索引（InnoDB）

## 主键和外键的区别

### 问题

主键和外键（从键）的区别是什么？

### 参考答案

| | 主键（Primary Key） | 外键（Foreign Key） |
|--|------|------|
| 作用 | 唯一标识本表中的每一行 | 引用另一张表的主键，建立关联关系 |
| 唯一性 | 必须唯一 | 可以重复 |
| 空值 | 不允许 NULL | 允许 NULL |
| 数量 | 每表只有一个 | 可以有多个 |
| 约束方向 | 约束本表数据 | 约束跨表引用关系 |

**示例**

```sql
CREATE TABLE users (
  id INT PRIMARY KEY,   -- 主键
  name VARCHAR(50)
);

CREATE TABLE orders (
  id INT PRIMARY KEY,
  user_id INT,          -- 外键，引用 users.id
  amount DECIMAL(10,2),
  FOREIGN KEY (user_id) REFERENCES users(id)
);
```

外键保证了：不能给一个不存在的用户创建订单。

**实际开发注意**：很多团队在高并发场景下不使用数据库层面的外键约束（影响写入性能、增加死锁风险），而是在应用层代码中保证引用完整性。

### 易错点

- 主键不一定是自增 ID，也可以是业务字段或联合主键
- 外键约束会影响 DELETE/UPDATE 操作（级联删除、置空等策略需要明确）
- 逻辑外键（不加约束）和物理外键（加 FOREIGN KEY 约束）是两回事

### 知识延伸

- 联合主键 vs 代理主键
- 外键的级联策略（CASCADE、SET NULL、RESTRICT）
- 分布式系统中的全局唯一 ID 生成方案（雪花算法等）
