# 浏览器缓存

---

## 一、缓存位置（优先级从高到低）

1. **Memory Cache（内存缓存）**：进程级，关闭 Tab 即失效，读取最快
2. **Disk Cache（磁盘缓存）**：持久化，容量大，跨会话有效
3. **Service Worker**：可编程缓存，由 JS 控制，可拦截请求
4. **Push Cache（HTTP/2）**：会话级，连接关闭即失效

一般来说：大文件走 Disk，小文件走 Memory；当前页面已加载的资源优先放 Memory。

---

## 二、强缓存

不发请求，直接用本地副本，状态码 `200 (from disk/memory cache)`。

### Expires（HTTP/1.0）

```
Expires: Wed, 21 Oct 2026 07:28:00 GMT
```

绝对时间，依赖客户端时钟，时钟不准就失效。

### Cache-Control（HTTP/1.1，优先级高于 Expires）

```
Cache-Control: max-age=3600, public, immutable
```

常见指令：

| 指令 | 含义 |
|---|---|
| `max-age=秒数` | 相对时间，更可靠 |
| `public` / `private` | 是否允许代理缓存 |
| `no-cache` | 跳过强缓存，每次走协商缓存 |
| `no-store` | 完全禁用缓存 |
| `immutable` | 内容永不变，连协商都省了 |
| `s-maxage` | 仅对代理服务器生效 |

---

## 三、协商缓存

强缓存失效后发请求，由服务器决定是否使用本地副本。命中返回 `304 Not Modified`，不带响应体。

### Last-Modified / If-Modified-Since

服务器首次返回 `Last-Modified: 时间`，客户端再次请求带上 `If-Modified-Since: 时间`。

缺点：

- 精度只到秒，1 秒内多次修改感知不到
- 文件内容没变但 mtime 变了（如 `git checkout`）会误判
- 分布式部署多台机器 mtime 可能不一致

### ETag / If-None-Match（优先级高于 Last-Modified）

服务器基于内容生成的指纹（如 hash），客户端再次请求带上 `If-None-Match: "abc123"`。

- 强 ETag：字节级一致
- 弱 ETag：`W/"abc123"`，语义一致即可

缺点：计算开销，分布式需要保证一致性。

---

## 四、完整流程

```
请求资源
  ↓
有强缓存且未过期？ → 是 → 直接用（200 from cache）
  ↓ 否
带上 If-None-Match / If-Modified-Since 发请求
  ↓
服务器对比
  ↓
未变 → 304，用本地副本
已变 → 200，返回新资源 + 新缓存头
```

---

## 五、实战策略

| 资源类型 | 推荐策略 | 说明 |
|---|---|---|
| HTML | `Cache-Control: no-cache` | 每次协商，保证入口能更新 |
| JS / CSS / 图片（带 hash） | `Cache-Control: max-age=31536000, immutable` | 一年强缓存，文件名变了自然换新 |
| API 接口 | `Cache-Control: no-store` | 避免缓存动态数据 |

## 六、高频追问

### Q1：刷新方式对缓存的影响？

- 普通刷新（F5）：跳过强缓存，走协商缓存
- 强制刷新（Ctrl+Shift+R）：跳过所有缓存，请求头会带 `Cache-Control: no-cache`
- 地址栏回车：所有缓存都用

### Q2：为什么用 hash 文件名 + 长强缓存？

解决"既要长缓存提速，又要能及时更新"的矛盾。内容变 → 文件名变 → URL 变 → 浏览器视为新资源去拉取。HTML 引用的是新文件名，所以入口只要不缓存就能拿到最新引用。

### Q3：no-cache 和 no-store 区别？

- `no-cache`：能缓存，但每次都要协商
- `no-store`：完全不缓存，最严格

### Q4：ETag 一定比 Last-Modified 好吗？

不一定。ETag 计算有成本，分布式场景下保证多机一致也麻烦。静态资源用 Last-Modified 通常够用，对一致性敏感的才上 ETag。

### Q5：客户端带的 If-Modified-Since 是哪个时间？

**就是上一次响应中服务器返回的 `Last-Modified` 的值，原样带回去。**

**首次请求**

```http
GET /style.css HTTP/1.1
```

**首次响应**

```http
HTTP/1.1 200 OK
Last-Modified: Wed, 13 May 2026 10:00:00 GMT
Content-Type: text/css

...文件内容...
```

浏览器把这个 `Last-Modified` 时间和资源一起存进缓存。

**再次请求**（强缓存过期后）

```http
GET /style.css HTTP/1.1
If-Modified-Since: Wed, 13 May 2026 10:00:00 GMT
```

把上次拿到的时间原封不动塞进 `If-Modified-Since`。

**服务器判断**

拿当前文件的修改时间和这个值比对：

- 没变 → `304 Not Modified`，无响应体
- 变了 → `200 OK` + 新内容 + 新的 `Last-Modified`

**关键点**

- 不是浏览器本地缓存的时间，也不是客户端当前时间
- 是服务器上次告诉浏览器的"资源最后修改时间"
- 所以这个时间基准在服务器，不受客户端时钟影响（这点比 `Expires` 强）

`ETag` / `If-None-Match` 同理：服务器给什么 ETag，下次请求就把它原样塞进 `If-None-Match`。

---

## 易错点

- `Expires` 用绝对时间、依赖客户端时钟，`Cache-Control: max-age` 用相对时间，更可靠
- `no-cache` 不是不缓存，而是"缓存了但每次都要协商"；`no-store` 才是真不缓存
- `Last-Modified` 精度是秒级，秒内多次修改无法识别
- 强缓存命中不发请求，所以服务端日志看不到这次访问
- `Cache-Control` 优先级高于 `Expires`，`ETag` 优先级高于 `Last-Modified`

## 知识延伸

- Service Worker + Cache API：可编程缓存，离线应用基础
- CDN 缓存层级（边缘节点 → 区域节点 → 源站）
- `Vary` 响应头：根据请求头（如 `Accept-Encoding`、`User-Agent`）区分缓存
- `stale-while-revalidate`：先用旧的，后台异步更新
