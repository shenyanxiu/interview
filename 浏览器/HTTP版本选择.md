# 如何选择 HTTP 版本

---

## 各版本核心特性对比

| 特性 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|---|---|---|
| 传输层 | TCP | TCP | QUIC（UDP） |
| 多路复用 | ❌ 一个连接一个请求 | ✅ 一个连接并行多个流 | ✅ 且无队头阻塞 |
| 头部压缩 | ❌ | ✅ HPACK | ✅ QPACK |
| 服务器推送 | ❌ | ✅（实际用得少） | ✅ |
| 连接建立 | TCP 三次握手 + TLS | TCP + TLS | 0-RTT / 1-RTT |
| 队头阻塞 | 应用层 + 传输层都有 | 解决了应用层，传输层仍有 | 彻底解决 |
| 兼容性 | 所有环境 | 主流浏览器和服务器 | 逐步普及中 |

---

## 各版本解决了什么问题

**HTTP/1.1 的痛点：**
- 队头阻塞：一个请求卡住，后面的都得等
- 浏览器对同一域名限制 6 个并发连接
- 头部冗余：每次请求都带完整 Header（Cookie 可能几 KB）
- 只能客户端发起请求

**HTTP/2 解决了：**
- 多路复用 — 一个 TCP 连接上并行传输多个请求/响应，不再需要域名分片
- 头部压缩 — HPACK 算法，增量编码
- 二进制分帧 — 更高效的解析
- 但仍有 TCP 层队头阻塞（一个包丢失，整个连接的所有流都等）

**HTTP/3 解决了：**
- 基于 QUIC（UDP），每个流独立，丢包只影响单个流
- 连接迁移 — 网络切换（WiFi → 4G）不断连
- 0-RTT 恢复连接 — 之前连过的服务器可以零往返建连

## 选择建议

### 直接上 HTTP/2（大多数场景的最优解）

适用条件：
- 网站已启用 HTTPS（HTTP/2 实际要求 TLS）
- 使用主流 Web 服务器（Nginx、Caddy、Cloudflare 等都默认支持）
- 面向现代浏览器用户

收益：
- 无需改代码，服务器配置开启即可
- 多路复用直接提升并发加载速度
- 可以去掉 HTTP/1.1 时代的优化 hack（雪碧图、域名分片、文件合并）

### 考虑 HTTP/3 的场景

- 用户网络环境差、丢包率高（移动端、弱网）
- 用户频繁切换网络（移动 App）
- 对首次连接速度极度敏感
- 已经用了 CDN（Cloudflare、AWS CloudFront 等已默认支持 HTTP/3）

### 仍需兼容 HTTP/1.1 的场景

- 内网/企业环境，老旧代理不支持 HTTP/2
- 客户端是老版本 curl、爬虫、IoT 设备
- 不使用 HTTPS 的内部服务（HTTP/2 理论支持明文但实际浏览器不支持）

## 实际部署策略

```
推荐做法：同时支持多版本，让客户端协商

客户端 ←→ CDN/负载均衡（HTTP/3 + HTTP/2）
CDN ←→ 源站（HTTP/2 或 HTTP/1.1 都行）
```

**Nginx 配置示例（HTTP/2）：**
```nginx
server {
    listen 443 ssl http2;
    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;
}
```

**协商机制：**
- HTTP/1.1 → HTTP/2：TLS 握手时通过 ALPN 扩展协商
- HTTP/2 → HTTP/3：服务端通过 `Alt-Svc` 响应头告知客户端支持 HTTP/3
  ```
  Alt-Svc: h3=":443"; ma=86400
  ```
- 客户端不支持高版本会自动降级，不会出错

---

## 一句话总结

| 场景 | 选择 |
|---|---|
| 通用 Web 应用 | HTTP/2（开启即用，零代码改动） |
| 移动端 / 弱网 / 高性能要求 | HTTP/3（通过 CDN 开启最省事） |
| 老旧客户端 / 内网 | 保留 HTTP/1.1 兼容 |
| 最佳实践 | 三个版本都支持，自动协商降级 |

## HTTP/2 和 HTTPS 的关系

**一句话：HTTP/2 协议本身不强制要求 HTTPS，但所有主流浏览器只支持基于 HTTPS 的 HTTP/2。**

### 协议规范 vs 实际实现

| | 规范（RFC 7540） | 实际情况 |
|---|---|---|
| HTTP/2 over TLS | 定义为 `h2` | ✅ 所有浏览器都支持 |
| HTTP/2 明文（无加密） | 定义为 `h2c` | ❌ 没有浏览器支持 |

理论上 HTTP/2 可以跑在明文 TCP 上（`h2c`），但浏览器厂商集体决定：不上 HTTPS 就不给用 HTTP/2。

### 为什么浏览器强制绑定 HTTPS？

1. **协商机制依赖 TLS** — 浏览器通过 TLS 握手中的 ALPN（Application-Layer Protocol Negotiation）扩展来协商使用 HTTP/2，没有 TLS 就没有这个协商通道
2. **推动全网加密** — 浏览器厂商借 HTTP/2 的性能优势倒逼网站升级 HTTPS
3. **安全考虑** — HTTP/2 的多路复用让一个连接承载更多数据，明文传输风险更大

### 实际影响

```
想用 HTTP/2？→ 必须先配好 HTTPS（SSL/TLS 证书）

没有 HTTPS → 浏览器自动降级到 HTTP/1.1
有 HTTPS    → 浏览器通过 ALPN 协商，能用 HTTP/2 就用
```

### 例外情况

- **内部服务间通信**（gRPC、微服务）可以用 `h2c`（明文 HTTP/2），因为不经过浏览器
- **curl、Postman** 等工具支持 `h2c`
- **反向代理内网段**：CDN → 源站之间可以用明文 HTTP/2

```
用户浏览器 ←— HTTPS + HTTP/2 —→ CDN/Nginx ←— h2c（明文）—→ 内部服务
                                  （必须加密）              （可以不加密）
```

### 总结

| 问题 | 答案 |
|---|---|
| HTTP/2 必须用 HTTPS 吗？ | 规范不强制，但浏览器强制 |
| 没有 HTTPS 能用 HTTP/2 吗？ | 浏览器不行，内部服务/工具可以（h2c） |
| HTTPS 就自动是 HTTP/2 吗？ | 不是，还需要服务器开启 HTTP/2 支持 |
| HTTP/3 也一样吗？ | 是的，HTTP/3 基于 QUIC，QUIC 内置了 TLS 1.3，加密是强制的 |

## HTTP 请求由谁发起？版本协商机制

### 谁发 HTTP 请求？

HTTP 请求由**客户端**发起，浏览器只是最常见的客户端之一。其他客户端：

- curl、Postman、HTTPie（命令行/工具）
- Node.js 的 fetch/axios（服务端发请求）
- 移动 App 的网络库（OkHttp、Alamofire）
- 爬虫、IoT 设备

### 版本协商是双方的事

最终用哪个 HTTP 版本，取决于**客户端和服务端都支持的最高版本**：

```
客户端支持 HTTP/2 + 服务端支持 HTTP/2 → 用 HTTP/2
客户端支持 HTTP/2 + 服务端只有 HTTP/1.1 → 降级到 HTTP/1.1
客户端只有 HTTP/1.1 + 服务端支持 HTTP/2 → 只能用 HTTP/1.1
```

浏览器必须支持对应版本才行。主流浏览器支持情况：

| 浏览器 | HTTP/2 | HTTP/3 |
|---|---|---|
| Chrome | 2015 年起 | 2020 年起 |
| Firefox | 2015 年起 | 2021 年起 |
| Safari | 2015 年起 | 2022 年起 |
| Edge | 2015 年起 | 2020 年起 |

### 协商过程

浏览器不是"选择"用哪个版本，而是通过协商机制自动决定：

**HTTP/1.1 → HTTP/2：**
- TLS 握手时，浏览器在 ClientHello 的 ALPN 扩展里列出自己支持的协议：`h2, http/1.1`
- 服务端从中选一个回复
- 整个过程对用户透明，不需要手动指定

**HTTP/2 → HTTP/3：**
- 浏览器先用 HTTP/2 正常请求
- 服务端在响应头里带 `Alt-Svc: h3=":443"`
- 浏览器记住这个信息，下次请求尝试用 HTTP/3（QUIC）
- 如果失败，自动回退到 HTTP/2

### 开发者需要关心什么？

**前端开发者：** 基本不用管，浏览器自动处理。`fetch()` 或 `XMLHttpRequest` 不需要指定 HTTP 版本。

**后端/运维：** 确保服务端开启了对应版本的支持，剩下的交给协商机制。

```javascript
// 前端代码完全一样，不管底层是 HTTP/1.1 还是 HTTP/2 还是 HTTP/3
const res = await fetch('https://api.example.com/users')
```
