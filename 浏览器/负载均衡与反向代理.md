# 负载均衡与反向代理

---

## 一、反向代理（Reverse Proxy）

### 一句话解释

反向代理是站在服务端前面的中间人，客户端以为在跟它通信，实际上它把请求转发给了后面的真实服务器。

### 正向代理 vs 反向代理

```
正向代理（代理客户端）：
客户端 → 代理服务器 → 目标服务器
         ↑ 客户端知道代理的存在，主动配置
         服务端不知道真实客户端是谁

反向代理（代理服务端）：
客户端 → 反向代理 → 真实服务器
         ↑ 客户端不知道后面有多少台服务器
         客户端以为反向代理就是服务器本身
```

| | 正向代理 | 反向代理 |
|---|---|---|
| 代理谁 | 客户端 | 服务端 |
| 谁知道它存在 | 客户端知道 | 客户端不知道 |
| 典型用途 | 翻墙、公司网络管控 | 负载均衡、安全防护 |
| 例子 | VPN、Shadowsocks | Nginx、Cloudflare |

### 反向代理能做什么

#### 1. 负载均衡

把请求分发到多台后端服务器：
```
用户 → Nginx → 服务器A / 服务器B / 服务器C
```

#### 2. 隐藏真实服务器

- 客户端只知道代理的 IP，不知道后端服务器的地址
- 后端服务器可以用内网 IP，不暴露在公网
- 攻击者无法直接攻击真实服务器

#### 3. SSL 终止（SSL Termination）

- 反向代理统一处理 HTTPS 加解密
- 后端服务器只需处理 HTTP 明文，减轻计算压力

```
用户 ←— HTTPS —→ Nginx ←— HTTP —→ 后端服务
                  （证书配在这里）    （不用管证书）
```

#### 4. 缓存静态资源

- 反向代理缓存图片、CSS、JS
- 相同请求直接返回缓存，不打到后端

#### 5. 压缩

- 统一做 gzip/brotli 压缩
- 后端服务不用关心压缩逻辑

#### 6. 跨域解决

- 前端和 API 不同源时，通过反向代理把 API 请求转成同源

```nginx
# 前端访问 /api/* 实际转发到后端服务
location /api/ {
    proxy_pass http://backend-server:3000/;
}
```

#### 7. 限流 / 安全防护

- 限制单 IP 请求频率
- 拦截恶意请求（WAF）
- 防 DDoS

### Nginx 配置示例

```nginx
server {
    listen 80;
    server_name www.example.com;

    # 静态资源直接返回
    location /static/ {
        root /var/www/;
        expires 30d;
    }

    # API 请求转发到后端
    location /api/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # 其他请求返回前端页面
    location / {
        root /var/www/dist;
        try_files $uri $uri/ /index.html;
    }
}
```

### 前端开发中最常接触的场景

开发环境代理（解决跨域），Vite / Webpack DevServer 本质就是一个反向代理：

```javascript
// vite.config.js
export default {
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:3000',
        changeOrigin: true
      }
    }
  }
}
```

浏览器请求 `localhost:5173/api/users` → DevServer 转发到 `localhost:3000/api/users` → 同源，没有跨域问题。

## 二、负载均衡（Load Balancing）

### 什么是负载均衡

把客户端请求分发到多台服务器，避免单台服务器过载，提高可用性和吞吐量。

```
客户端 → 负载均衡器 → 服务器 A
                    → 服务器 B
                    → 服务器 C
```

### 负载均衡的层级

| 层级 | 工作在 | 代表 | 特点 |
|---|---|---|---|
| L4（传输层） | TCP/UDP | LVS、AWS NLB | 只看 IP + 端口，转发快，不解析内容 |
| L7（应用层） | HTTP | Nginx、HAProxy、AWS ALB | 能看 URL/Header/Cookie，可做精细路由 |
| DNS 级别 | DNS 解析 | CloudFlare、Route53 | 按地域/权重返回不同 IP |

**实际架构常见组合：**

```
DNS 负载均衡（地域分流）
  → L4 负载均衡（高性能转发）
    → L7 负载均衡（精细路由）
      → 应用服务器集群
```

### 常见负载均衡算法

#### 1. 轮询（Round Robin）

依次分配，最简单：

```
请求1 → A, 请求2 → B, 请求3 → C, 请求4 → A ...
```

适用：服务器配置相同、请求处理时间接近。

#### 2. 加权轮询（Weighted Round Robin）

按权重分配，性能好的多分点：

```
A(权重5)  B(权重3)  C(权重2)
10个请求 → A拿5个, B拿3个, C拿2个
```

适用：服务器配置不同。

#### 3. IP Hash

同一客户端 IP 始终打到同一台服务器：

```
hash(客户端IP) % 服务器数量 = 目标服务器
```

适用：需要会话保持（Session 粘性），但服务器挂了会丢失会话。

#### 4. 最少连接数（Least Connections）

把请求分给当前连接数最少的服务器：

```
A(当前10连接)  B(当前3连接)  C(当前7连接)
新请求 → B
```

适用：请求处理时间差异大（长连接场景）。

#### 5. 加权最少连接

结合权重和连接数：`连接数 / 权重` 最小的优先。

#### 6. 一致性哈希（Consistent Hashing）

把服务器和请求都映射到哈希环上，请求顺时针找到最近的服务器。增减节点只影响相邻区间，不会导致大规模重新分配。

适用：缓存场景（Redis 集群、CDN 节点）。

### 健康检查

负载均衡器需要知道后端服务器是否存活：

**被动检查：**
- 转发请求后发现超时/5xx → 标记为不健康
- 连续失败 N 次后摘除

**主动检查：**
- 定期发探针请求（如 `GET /health`）
- 服务器返回 200 → 健康
- 超时或非 200 → 不健康，从池中摘除
- 恢复后自动加回

```nginx
# Nginx 健康检查配置示例
upstream backend {
    server 192.168.1.1:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.2:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.3:8080 backup;  # 备用节点
}
```

### 会话保持（Session Affinity）

问题：用户登录后 Session 存在服务器 A，下次请求被分到服务器 B 就丢失登录状态。

| 方案 | 做法 | 优缺点 |
|---|---|---|
| IP Hash | 同 IP 固定到同一台 | 简单，但 NAT 后多用户共享 IP |
| Cookie 粘性 | 负载均衡器种 Cookie 标记目标服务器 | 灵活，但服务器挂了仍丢失 |
| Session 共享 | Session 存 Redis/数据库 | 根本解决，推荐 |
| JWT 无状态 | Token 自包含用户信息，不依赖服务端 Session | 最佳实践 |

### 常见负载均衡器对比

| 工具 | 层级 | 特点 |
|---|---|---|
| Nginx | L7 | 最常用，兼做反向代理/静态资源/缓存 |
| HAProxy | L4/L7 | 高性能，配置灵活，监控面板好 |
| LVS | L4 | Linux 内核级，性能极高，功能简单 |
| AWS ALB | L7 | 托管服务，支持路径/Host 路由 |
| AWS NLB | L4 | 托管服务，超高吞吐，固定 IP |
| Cloudflare | DNS/L7 | CDN + 负载均衡一体 |

## 三、灰度发布（Canary Release）

### 什么是灰度发布

新版本不一次性全量上线，而是先让一小部分用户用新版本，观察没问题后再逐步扩大到所有用户。

```
全部用户流量（100%）
  ├── 95% → 旧版本（稳定）
  └── 5%  → 新版本（观察中）

观察没问题 → 逐步调整为 10% → 30% → 50% → 100%
发现问题   → 立刻切回 0%，全部走旧版本
```

### 流量分配策略

| 策略 | 做法 | 场景 |
|---|---|---|
| 按比例随机 | 5% 的请求随机打到新版本 | 最常见 |
| 按用户 ID | 尾号为 0 的用户走新版本 | 保证同一用户体验一致 |
| 按地域 | 先在某个城市上线 | 区域性功能 |
| 按 Cookie/Header | 带特定标记的走新版本 | 内部测试 |
| 白名单 | 只有指定用户能看到新版本 | 产品验收、内测 |

### Nginx 灰度配置示例

```nginx
upstream stable {
    server 192.168.1.1:8080;
    server 192.168.1.2:8080;
}

upstream canary {
    server 192.168.1.3:8080;  # 新版本
}

# 带特定 header 的走灰度
map $http_x_gray $backend {
    "true"  canary;
    default stable;
}

server {
    location / {
        proxy_pass http://$backend;
    }
}
```

### 相关概念区分

| 术语 | 含义 |
|---|---|
| 灰度发布 | 按比例逐步放量 |
| 蓝绿部署 | 两套完整环境，一键切换流量（要么全走蓝，要么全走绿） |
| A/B 测试 | 类似灰度，但目的是对比数据指标（转化率、点击率），选出更优方案 |
| 金丝雀发布 | 灰度发布的别名（来源：矿井里用金丝雀探测毒气） |

## 四、面试常见追问

**Q：Nginx 和 LVS 的区别？**
- LVS 工作在内核态（L4），不解析 HTTP，转发性能极高（百万级并发）
- Nginx 工作在用户态（L7），能解析 HTTP 做精细路由，但性能低于 LVS
- 大型架构：LVS 在前做粗粒度分发，Nginx 在后做细粒度路由

**Q：负载均衡器本身挂了怎么办？**
- 主备模式（Keepalived + VIP）：主节点挂了，备节点秒级接管虚拟 IP
- 云服务自带高可用（ALB/NLB 多可用区部署）

**Q：灰度发布怎么做？**
- L7 负载均衡根据 Header/Cookie/IP 把部分流量导到新版本
- 例如：`X-Gray: true` 的请求打到灰度服务器

**Q：反向代理和负载均衡的关系？**
- 反向代理是一种架构模式（代理服务端）
- 负载均衡是反向代理的一个功能（把请求分发到多台服务器）
- Nginx 既是反向代理，也是负载均衡器
