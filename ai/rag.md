# RAG（Retrieval-Augmented Generation）

## Q1: RAG 是什么，怎么用？

### 问题

RAG 是什么？它的工作流程是怎样的？

### 主题

AI / LLM 应用架构

### 难度

中级

### 参考答案

**定义**：RAG（检索增强生成）是一种让 LLM 在生成回答前，先从外部知识库中检索相关信息，再结合检索结果生成答案的技术架构。核心思路是"先搜再答"。

**解决的问题**：
- 知识截止：训练数据有时间边界
- 幻觉：模型会编造不存在的内容
- 私有数据：无法访问企业内部文档
- 上下文有限：不可能把所有知识塞进 prompt

**工作流程**：

```text
用户提问
   ↓
① Query 处理（改写/拆分/向量化）
   ↓
② 检索（从向量数据库/全文索引中召回相关文档片段）
   ↓
③ 重排序（对召回结果按相关性排序，取 Top-K）
   ↓
④ 组装 Prompt（检索到的上下文 + 用户问题拼成 prompt）
   ↓
⑤ LLM 生成（基于增强后的 prompt 生成回答）
   ↓
返回答案（可附带引用来源）
```

**核心组件**：

1. **文档处理（离线）**：原始文档 → 分块(Chunking) → 向量化(Embedding) → 存入向量数据库
2. **检索（在线）**：向量检索（语义相似度）、关键词检索（BM25）、混合检索
3. **生成**：将检索结果注入 prompt，让 LLM 基于上下文生成答案

**关键优化点**：
- 分块策略：200-500 tokens，带重叠（overlap 50-100 tokens）
- Query 改写：HyDE（假设性答案检索）、多查询、Step-back
- 重排序：用 Cross-Encoder 对召回结果精排

**与微调的对比**：
- RAG：知识更新快、成本低、可追溯来源、适合知识密集型问答
- Fine-tuning：需重新训练、成本高、适合风格/格式/推理能力调整

### 追问链

- 分块策略怎么选？太大太小各有什么问题？
- HyDE 是什么原理？
- RAG 和 Fine-tuning 什么时候该用哪个？
- 如何评估 RAG 系统的效果？

### 易错点

- 以为 RAG 能完全消除幻觉（只能减少，不能杜绝）
- 忽略分块质量对检索效果的影响
- 只用向量检索而忽略关键词检索的互补作用

### 知识延伸

- GraphRAG：用知识图谱增强检索，处理多跳推理
- Agentic RAG：Agent 自主决定何时检索、检索什么、是否需要二次检索
- 多模态 RAG：支持图片、表格等非文本内容的检索

## Q2: RAG 一定要向量数据库吗？

### 问题

RAG 的检索环节是否必须使用向量数据库？

### 主题

AI / RAG 架构选型

### 难度

中级

### 参考答案

**不一定。** 向量数据库只是 RAG 检索环节的一种实现方式，不是必要条件。

**向量存储的多种选择**：

| 方案 | 适用场景 |
|------|----------|
| 专用向量数据库（Milvus、Pinecone） | 大规模、高性能、生产环境 |
| PostgreSQL + pgvector | 已有 PG，数据量中等 |
| SQLite + 向量扩展 | 本地/轻量应用 |
| 内存（FAISS） | 数据量小、原型验证 |
| 直接文件存储 | 几百条数据，暴力计算即可 |

**其他检索方式**：
- 关键词检索（ES/BM25）：精确匹配强，对代码符号友好
- 混合检索：向量 + 关键词结合，生产环境最常见
- 知识图谱：适合多跳推理
- 直接塞上下文：文档量小时，全部放进 prompt 最简单

**选型建议**：

```text
数据量 < 几十篇文档     → 直接塞进 prompt
数据量适中 + 已有 PG    → pgvector 扩展
需要精确匹配（代码/API） → 关键词检索
通用问答场景            → 混合检索
大规模 + 高并发         → 专用向量数据库
```

### 追问链

- pgvector 和专用向量数据库性能差距有多大？
- 混合检索中向量和关键词结果怎么融合排序？
- FAISS 和 Milvus 的区别是什么？

### 易错点

- 小项目过度工程化，几十篇文档就上 Milvus
- 只用向量检索，对精确关键词（错误码、函数名）匹配差
- 忽略"直接塞上下文"这个最简单的方案

## Q3: RAG 的检索方式是否不唯一？

### 问题

RAG 的检索方式有哪些？是否可以用 Web Search、API 调用等多种方式？

### 主题

AI / RAG 架构设计

### 难度

中级

### 参考答案

**完全正确，方式不唯一。** RAG 中的 "R"（Retrieval）只规定了"要检索"，没规定"怎么检索"。

**常见检索方式**：

| 检索方式 | 知识来源 | 典型场景 |
|----------|----------|----------|
| Web Search | 互联网实时内容 | 时事、最新文档版本 |
| 向量数据库 | 预处理好的私有文档 | 企业知识库问答 |
| 关键词搜索（ES/BM25） | 结构化/半结构化数据 | 代码搜索、日志查询 |
| SQL 查询 | 关系型数据库 | "上个月销售额多少" |
| API 调用 | 第三方服务 | 天气、股票、内部系统 |
| 文件系统读取 | 本地文件 | IDE 中读取项目代码 |
| 知识图谱遍历 | 图数据库 | 多跳关系推理 |

**可以组合使用**：一个问题同时触发多种检索方式，合并结果后送给 LLM。

**本质抽象**：

```text
问题 → 【用任意方式获取相关上下文】 → 上下文 + 问题 → LLM → 答案
```

中间的检索方式完全取决于数据在哪、是什么形态。向量数据库只是因为对"非结构化文本的语义搜索"特别好用，所以被提得最多，但它不是 RAG 的定义本身。

### 追问链

- 多种检索方式的结果如何合并和排序？
- Agentic RAG 中 Agent 如何决定用哪种检索方式？
- Web Search 作为检索源时，如何控制信息质量？

### 易错点

- 把 RAG 等同于"向量数据库 + LLM"
- 忽略结构化数据（SQL、API）也可以作为检索源
- 没有意识到多种检索方式可以组合使用

### 知识延伸

- Agentic RAG：Agent 自主选择检索策略
- Tool Use / Function Calling：LLM 调用外部工具获取信息，本质也是一种 RAG
- 多源融合排序（Reciprocal Rank Fusion）：合并多路检索结果的常用算法
