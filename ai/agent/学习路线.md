# Agent 开发学习路线

## 主题

AI Agent 开发

## 适用人群

前端 + Node.js 背景，想系统学习 Agent 开发的工程师

## 学习路线

### 一、核心概念层（吃透原理）

目标：能解释"为什么这样设计"，是面试和技术决策的基础。

- **LLM 基础认知** — token、context window、temperature、流式输出（SSE）、上下文截断与压缩
- **Function Calling / Tool Use** — OpenAI / Anthropic 两家的协议差异，JSON Schema 怎么描述工具
- **ReAct 模式** — Reason + Act 循环，Thought / Action / Observation 三段式
- **Agent Loop** — 一轮 Agent 的完整执行流程：意图理解 → 规划 → 调用工具 → 观察结果 → 继续/终止
- **Prompt Engineering** — System Prompt、Few-shot、CoT（思维链）、结构化输出
- **记忆机制** — 短期记忆（对话历史）、长期记忆（向量数据库 / RAG）

### 二、协议与生态

- **MCP 协议** — Transport 层（stdio vs HTTP SSE）、Tool / Resource / Prompt 三类能力、动手写一个 MCP Server
- **A2A（Agent-to-Agent）** — Google 提的多 Agent 通信协议，了解概念即可
- **OpenAPI / JSON Schema** — Tool 定义底层都是它，前后端 API 设计经验直接复用

### 三、主流框架（挑一个深入用）

不用全学，挑一个动手做项目：

- **Vercel AI SDK** — TS 友好，前端集成最顺，**推荐优先**
- **LangChain / LangGraph**（JS 版有 LangChain.js）— 生态最大，概念繁杂，适合学习全貌
- **Mastra** — 新兴 TS Agent 框架，设计现代
- **CrewAI / AutoGen** — 多 Agent 协作场景

### 四、工程化能力（拉开差距的地方）

做完 Demo 之后真正要花时间的部分：

- **流式响应** — SSE / WebSocket，前端边出边渲染（Markdown 流式渲染、打字机效果）
- **工具调用编排** — 串行、并行、条件分支、循环终止条件、超时与重试
- **错误处理** — LLM 输出格式错误、工具调用失败、token 超限、幻觉检测
- **可观测性** — Trace（每一步推理和调用都可追踪）、LangSmith / Langfuse 这类工具
- **成本与性能** — token 计费、缓存策略（Prompt Caching）、模型路由（小模型干小活）
- **评估（Eval）** — 怎么判断 Agent 输出对不对，golden set、LLM-as-Judge
- **安全** — Prompt Injection 防护、工具权限隔离、敏感信息脱敏

### 五、RAG 与知识库

Agent 经常要"查资料"，独立成一个领域：

- 文档切分（Chunking）策略
- Embedding 模型选型（OpenAI / BGE / 国产）
- 向量数据库（Pinecone / Milvus / pgvector）
- 检索策略（向量 + 关键词混合、Rerank）
- RAG 评估指标（召回率、相关性）

### 六、动手项目建议

按这个顺序练手，每一步比上一步复杂一档：

1. **基础对话 Agent** — Vercel AI SDK 写一个带工具调用的聊天框，工具就两个：查天气、查日期
2. **写一个 MCP Server** — 比如把 interview 笔记目录暴露成 MCP，让 Claude / Kiro 能搜索答题
3. **代码 Agent** — 给一个 GitHub Issue，让 Agent 读代码、定位、改代码、跑测试
4. **多 Agent 协作** — 比如"产品经理 Agent + 开发 Agent + 测试 Agent"协同完成需求
5. **带 RAG 的领域 Agent** — 把企业文档喂进去做问答

## 推荐学习顺序（前端 + Node 路线）

```
Function Calling 原理
   ↓
Vercel AI SDK 实战
   ↓
MCP Server 开发
   ↓
流式 / 工程化
   ↓
RAG
   ↓
多 Agent 协作
```

## 易错点

- 一上来就学框架，跳过 Function Calling / ReAct 等底层原理，遇到问题不知道怎么调
- 只做 Demo 不做工程化（流式、错误处理、评估、可观测性），生产环境直接翻车
- 把 RAG 当万能解法，忽略了 Prompt Engineering 和上下文管理本身的优化空间
- 过早引入多 Agent 架构，单 Agent 都没调好就追求复杂编排
