# 前后端通信分层架构（JSON 对外，gRPC 对内）

## 问题

为什么浏览器调用后端用 JSON，而服务器之间用 gRPC？这种分层架构是怎么设计的？

## 主题

架构设计 / 协议选型

## 难度

中级

## 参考答案

这是当前主流互联网公司的**典型分层通信架构**："对外用 JSON 求兼容，对内用 Protobuf 求性能"。

### 经典架构图

```
┌─────────────┐
│   浏览器    │  ← 用户终端
│  / App端    │
└──────┬──────┘
       │
       │ HTTP/HTTPS + JSON（REST / GraphQL）
       │ 可读、通用、调试方便
       │
       ▼
┌─────────────────────────────┐
│   网关层 / BFF 层            │  ← 协议转换枢纽
│  （Nginx / Kong / 自研）     │
└──────┬──────────────────────┘
       │
       │ gRPC + Protobuf（内部高性能通信）
       │ 小、快、强契约
       │
       ▼
┌────────┬────────┬────────┬────────┐
│ 用户   │ 订单   │ 支付   │ 库存   │  ← 后端微服务
│ 服务   │ 服务   │ 服务   │ 服务   │
└────────┴────────┴────────┴────────┘
```

### 为什么这样分层

#### 1. 浏览器/App ↔ 后端：用 JSON

| 原因 | 说明 |
|------|------|
| **浏览器原生支持** | `JSON.parse / stringify` 内置，无需依赖 |
| **可读可调试** | F12 Network 面板一眼能看懂 |
| **生态成熟** | Postman、curl、Swagger 全家桶 |
| **跨端通用** | Web、iOS、Android、小程序统一接入 |
| **第三方友好** | 开放 API 必须用 JSON |

#### 2. 服务 ↔ 服务：用 gRPC + Protobuf

| 原因 | 说明 |
|------|------|
| **性能优先** | 内部 QPS 高，省 1ms 都重要 |
| **强类型契约** | `.proto` 文件即文档，避免对接扯皮 |
| **跨语言** | Java + Go + Node 混搭无压力 |
| **流式调用** | 双向流原生支持 |
| **不需要人读** | 反正机器对机器通信 |

### BFF / 网关层的作用

中间这一层是**协议转换的关键**，常见职责：

```
浏览器 JSON ──► BFF ──► gRPC ──► 微服务
浏览器 JSON ◄── BFF ◄── gRPC ◄── 微服务
```

- **协议转换**：HTTP+JSON ↔ gRPC+Protobuf
- **聚合裁剪**：把多个微服务的数据聚合成前端需要的形状
- **认证鉴权**：JWT 校验、限流、防刷
- **跨域处理**：CORS
- **日志监控**：统一埋点

**典型实现**：
- Node.js BFF（前端团队最爱）
- Spring Cloud Gateway
- Kong / APISIX / Envoy
- gRPC-Gateway（自动把 gRPC 转 REST）

### 几种常见模式

#### 模式一：标准分层（最常见）
```
浏览器 ──JSON──► BFF ──gRPC──► 微服务
```
适合中大型团队，前后端解耦。

#### 模式二：gRPC-Web（直连）
```
浏览器 ──gRPC-Web──► 微服务
```
浏览器直接用 gRPC，需要代理转换 HTTP/2 帧，适合内部系统。

#### 模式三：纯 REST（小团队）
```
浏览器 ──JSON──► 单体后端
浏览器 ──JSON──► 微服务（互相也用 HTTP+JSON）
```
简单粗暴，性能够用就行。

#### 模式四：GraphQL 网关
```
浏览器 ──GraphQL──► 网关 ──gRPC──► 微服务
```
前端按需取数据，灵活但复杂。

### 协议选择速查表

| 通信链路 | 推荐协议 | 序列化 |
|----------|----------|--------|
| 浏览器 ↔ 后端 | HTTP / HTTPS | JSON |
| App ↔ 后端 | HTTP / HTTPS | JSON / Protobuf |
| 服务 ↔ 服务（同机房） | gRPC | Protobuf |
| 服务 ↔ 服务（跨机房） | gRPC | Protobuf |
| 服务 ↔ 消息队列 | Kafka / RocketMQ | Protobuf / JSON |
| 服务 ↔ 数据库 | MySQL / Redis 协议 | 各自二进制 |
| 实时推送 | WebSocket / SSE | JSON |
| 浏览器 ↔ 内部服务 | gRPC-Web（少见） | Protobuf |

### App 端的演进趋势

虽然浏览器主要用 JSON，但**移动 App 端**正在从 JSON 转向 Protobuf：

- 流量更省（用户流量贵）
- 电量更省（解析快）
- 弱网更稳

微信、王者荣耀、抖音的核心协议都是 Protobuf，但它们的**开放 API**（给第三方调用的）依然是 JSON。

### 真实公司架构示例

- **字节跳动 / 阿里 / 腾讯**：Web/App ──HTTP+JSON──► API Gateway ──gRPC/Thrift+PB──► 几千个微服务
- **Google**：浏览器 ──HTTPS+JSON──► Frontend Server ──Stubby(gRPC前身)──► 后端服务
- **美团**：对外 HTTP + JSON，内部 gRPC / Thrift + Protobuf，消息队列 Kafka + Protobuf

### 核心原则

> **对外用 JSON 求兼容，对内用 Protobuf 求性能。**

中间靠 BFF / 网关做协议翻译，前端开发者通常只感知 JSON，后端开发者两种都要熟悉。

## 易错点

- 全部使用 JSON，导致内部通信性能瓶颈
- 全部使用 Protobuf，导致浏览器端集成困难
- BFF 层职责膨胀，把核心业务逻辑放进去
- 没有统一的网关层，认证 / 限流 / 监控分散

## 知识延伸

- BFF（Backend For Frontend）模式
- API Gateway 选型（Kong / APISIX / Envoy）
- gRPC-Gateway：自动生成 REST 接口的工具
- Service Mesh（Istio）与传统网关的关系
- Protobuf Schema 在多团队协作中的版本管理
