# 服务器之间通信的方式

## 问题

服务器之间通信的方式有哪些？各自适用什么场景？

## 主题

架构设计 / 分布式系统

## 难度

中级

## 参考答案

服务器间通信总体可以分为五大类：**网络协议直连、RPC、消息队列、共享存储、服务网格**。底层都是 TCP/UDP，区别在于上层抽象方式。

### 一、基于网络协议的通信

#### 1. TCP / UDP（传输层）
- **TCP**：面向连接、可靠、有序，适合数据完整性要求高的场景（文件传输、数据库同步）
- **UDP**：无连接、不可靠但低延迟，适合实时场景（音视频、游戏、心跳包）
- 通常通过 Socket 编程直接操作

#### 2. HTTP / HTTPS
- 最通用的应用层协议，基于 TCP，无状态、跨语言
- RESTful API 是最常见的微服务间通信方式
- 缺点：头部冗余、性能不如二进制协议

#### 3. WebSocket
- 全双工长连接，建立在 HTTP 之上后协议升级
- 适合服务端主动推送、实时通信（聊天、股票行情）

### 二、RPC（远程过程调用）

让远程调用看起来像本地函数调用，性能优于 HTTP。

| 框架 | 协议 | 序列化 | 特点 |
|------|------|--------|------|
| **gRPC** | HTTP/2 | Protobuf | 跨语言、流式、高性能（Google） |
| **Thrift** | TCP | Thrift IDL | 跨语言（Facebook） |
| **Dubbo** | TCP/HTTP | Hessian/Protobuf | Java 生态、阿里出品 |
| **JSON-RPC** | HTTP | JSON | 简单轻量 |

核心要素：**序列化 + 网络传输 + 服务发现 + 负载均衡**

### 三、消息队列（异步通信）

服务通过中间件解耦，不直接依赖。

- **Kafka**：高吞吐、持久化、日志/大数据流处理
- **RabbitMQ**：AMQP 协议、路由灵活、业务消息
- **RocketMQ**：阿里出品，事务消息能力强
- **Redis Stream / Pub/Sub**：轻量级发布订阅

**优势**：异步、削峰、解耦、最终一致性
**劣势**：增加系统复杂度，需处理重复消费、消息丢失

### 四、共享存储（间接通信）

- **数据库**（MySQL、PostgreSQL）：通过共享表交换数据
- **缓存**（Redis、Memcached）：共享状态、分布式锁
- **文件系统**（NFS、对象存储 S3）：大文件交换

不算严格的"通信"，但分布式系统中常用作数据共享通道。

### 五、服务网格 / 高级模式

- **Service Mesh**（Istio、Linkerd）：通过 Sidecar 代理处理服务间通信，业务无感知
- **GraphQL Federation**：多个服务联邦成一个 GraphQL 网关
- **Event-Driven / EventBus**：事件驱动架构，配合消息队列

### 选型建议

| 场景 | 推荐方式 |
|------|----------|
| 对外暴露 API | HTTP / HTTPS（REST、GraphQL） |
| 内部微服务高频调用 | gRPC、Dubbo |
| 异步解耦、削峰 | Kafka、RocketMQ |
| 实时推送 | WebSocket、SSE |
| 跨语言 RPC | gRPC、Thrift |
| 简单数据共享 | Redis、数据库 |

### 核心思路

**同步用 RPC/HTTP，异步用 MQ，实时用 WebSocket，底层都基于 TCP/UDP。**

选型看四个维度：
1. 性能要求
2. 一致性要求
3. 耦合度
4. 跨语言需求

## 易错点

- 把所有通信都用 HTTP+JSON，忽视性能成本
- 滥用消息队列，把同步业务也异步化，导致一致性问题
- 忽略服务发现和负载均衡，硬编码 IP
- 没有重试 / 超时 / 熔断机制，一个服务挂掉级联雪崩
