# 工程化与协作面试题

## Q1：Git Hook + LLM 流式翻译的中英双语同步流水线怎么设计的？

### 整体方案

1. **触发时机**：在 `pre-push` 或 CI 阶段检测中文文档变更（`git diff --name-only` 过滤 `.md` 文件）
2. **增量翻译**：只对 diff 中变更的段落调用 LLM 翻译，非全量重翻
3. **流式输出**：使用 SSE/stream 模式调用 LLM API，边生成边写入英文文件，避免长文档超时
4. **结果对齐**：中英文件保持相同的目录结构和文件名映射（如 `docs/zh/` → `docs/en/`）
5. **质量保障**：翻译结果自动 commit 到同一个 PR，人工 review 后合并

### 追问1：增量 diff 怎么做的？

```bash
# pre-push hook 中
changed_files=$(git diff --name-only HEAD~1 HEAD -- docs/zh/**/*.md)
```

仅知道哪些文件变了还不够，还需要知道文件内哪些段落变了：

```javascript
const oldContent = getGitFileContent(file, 'HEAD~1');
const newContent = fs.readFileSync(file, 'utf-8');

// 按段落（## 标题）切分
const oldSections = splitBySections(oldContent);
const newSections = splitBySections(newContent);

// diff 出变更的段落
const changedSections = diffSections(oldSections, newSections);

// 只翻译变更的段落
for (const section of changedSections) {
  const translated = await streamTranslate(section);
  updateEnglishFile(file, section.id, translated);
}
```

好处：
- 一篇 3000 字的文档只改了一个段落，只翻译 200 字，节省 90%+ token
- 未变更段落保持之前人工校对过的翻译结果，质量更稳定

### 追问2：流式翻译的"流式"体现在哪里？

```javascript
async function streamTranslate(text) {
  const response = await fetch(LLM_API, {
    method: 'POST',
    body: JSON.stringify({
      messages: [{ role: 'user', content: `翻译为英文：\n${text}` }],
      stream: true
    })
  });

  let result = '';
  const reader = response.body.getReader();
  const decoder = new TextDecoder();
  
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    const chunk = decoder.decode(value);
    const content = parseSSEChunk(chunk);
    result += content;
    process.stdout.write(content); // 实时输出进度
  }
  
  return result;
}
```

流式的意义：
- **避免超时**：长文档翻译可能耗时 30s+，非流式请求可能被网关超时断开
- **可观察性**：CI 流水线中可以实时看到翻译进度
- **失败恢复**：如果中途断开，已经收到的部分可以保留，只重试剩余部分

### 追问3：翻译质量怎么保障？术语一致性问题？

1. **术语表注入 prompt**：
```
以下术语请保持统一翻译：
- 快应用 → Quick App
- 组件 → Component  
- 生命周期 → Lifecycle
- 真机调试 → On-device Debugging
请翻译以下内容：...
```

2. **后处理校验**：翻译完后扫描是否有术语表中的中文残留或不一致翻译
3. **人工 review 兜底**：自动翻译结果 commit 到 PR，技术写作同学 review 后合并

## Q2：你是怎么推动接口规范标准化的？

### 统一响应格式

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "list": [],
    "total": 100,
    "page": 1,
    "pageSize": 20
  }
}
```

### 错误码分段

| 范围 | 模块 |
|------|------|
| 1000-1999 | 通用错误（参数校验、鉴权） |
| 2000-2999 | 播放器业务 |
| 3000-3999 | 相册业务 |
| 4000-4999 | 应用商店 |

### 推动落地方式

- **文档先行**：接口变更必须先更新 API 文档，PR 里没有文档更新不给合
- **接口评审**：新接口上线前拉前后端一起 review，确认字段命名、类型、是否可选
- **前端封装**：在公共请求层做响应拦截，非规范格式直接报警
- **实际收益**：减少了联调阶段的来回沟通成本，新人接手项目也更容易理解接口含义
- 结合多端场景（手表/音箱/App），明确哪些字段各端必须、哪些可选

## Q3：30+ 款应用适配是怎么推动的？遇到三方不配合怎么办？

- 建立标准化的适配流程文档和 checklist
- 分优先级推进：先推 DAU 高的核心应用，再推长尾应用
- 遇到不配合的情况：
  - 明确适配收益（装机量数据）
  - 给出 SDK/模板降低对方工作量
  - 必要时升级到 BD 层面推动
