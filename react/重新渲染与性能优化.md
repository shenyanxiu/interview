# React 重新渲染与性能优化

> 面试笔记：围绕「React 何时重新渲染」以及如何用 `React.memo` / `useMemo` 控制不必要的渲染。

---

## Q1：React 在什么情况下会重新渲染？

- **主题**：React
- **难度**：中级

### 参考答案

React 的「重新渲染」指的是重新执行函数组件（或类组件的 `render`），生成新的虚拟 DOM 并参与 diff，**不等于真实 DOM 一定会更新**。触发条件主要有 5 类：

1. **自身状态变化**
   - `useState` / `useReducer` 调用 setter
   - 类组件的 `setState` / `forceUpdate`
   - React 用 `Object.is` 比较新旧值，相同则 bail out 跳过渲染

2. **父组件重新渲染**
   - 默认情况下，父组件一旦渲染，所有子组件都会跟着渲染（无论 props 是否变化）
   - 这是「无效渲染」最常见的来源

3. **订阅的 Context 变化**
   - 通过 `useContext` 订阅了某个 Context 的组件，在 Provider 的 `value` 变化（`Object.is` 不等）时一定重新渲染
   - 直接传字面量对象 / 数组容易导致下游全部重渲染

4. **订阅的外部数据源变化**
   - 通过 `useSyncExternalStore` 订阅的外部 store（Redux、Zustand 底层都基于此）更新，且选择器返回值变化

5. **副作用中再次更新状态**
   - `useEffect` / `useLayoutEffect` 里 `setState`，会在当前渲染完成后触发新的渲染

### 不会触发渲染的情况

| 操作 | 是否渲染 |
|---|---|
| 直接修改 state（`state.count++`） | 否（React 检测不到） |
| `useRef` 的 `.current` 变化 | 否 |
| 普通变量赋值 | 否 |
| `setState` 传入 `Object.is` 相等的值 | 否（bail out） |

### 易错点

- 把「重新渲染」等同于「真实 DOM 更新」。重新渲染只是重新执行组件函数，是否更新 DOM 取决于 diff 结果。
- 认为 props 没变子组件就不会渲染。**默认会渲染**，需要 `React.memo` 才能浅比较 props。
- 直接 mutate state（`arr.push(x); setArr(arr)`），引用没变，React 不会重新渲染。
- Context 的 `value` 直接写成字面量 `{}` / `[]`，每次渲染都是新对象，下游全量重渲。

### 可能的追问

- 父组件渲染时，子组件如何避免跟着渲染？→ `React.memo` + 稳定引用
- `Object.is` 和 `===` 有什么区别？→ 主要差在 `NaN` 和 `+0/-0`
- 重新渲染和真实 DOM 更新的关系？→ vDOM diff 决定是否落到真实 DOM
- 为什么 `useState` 传同样的值不会渲染，但 `setState` 会？→ Hook 有 bail out 优化，类组件没有
- 如何判断一次渲染是否必要？→ React DevTools Profiler，看「Why did this render?」

### 知识延伸

- React Fiber 的渲染流程：render 阶段 → commit 阶段
- Concurrent Mode 下的 lane 调度与可中断渲染
- StrictMode 在开发环境会双调用组件函数，用来暴露副作用问题（不是重渲）
- 状态提升与状态下放对渲染范围的影响

## Q2：React.memo 和 useMemo 是怎么优化的？区别在哪？

- **主题**：React
- **难度**：中级

### 参考答案

两者都用「空间换时间」做缓存，但缓存的对象不同：

| 维度 | `React.memo` | `useMemo` |
|---|---|---|
| 缓存对象 | 整个组件的渲染结果 | 单个值 / 引用 |
| 比较内容 | props（默认浅比较） | 依赖数组（`Object.is` 逐项比较） |
| 用在哪 | 包裹组件定义 | 组件函数体内 |
| 单独使用是否有效 | props 含引用类型时常失效 | 总是生效 |

#### React.memo

- 包裹组件，让它在 props 浅比较相等时**跳过重新渲染**，复用上次结果
- 用来打破「父组件渲染必定带动子组件渲染」的默认行为
- 可传第二个参数自定义比较函数：`React.memo(Comp, (prev, next) => boolean)`

```jsx
const Child = React.memo(function Child({ name }) {
  return <div>{name}</div>;
});
```

#### useMemo

- 缓存一个计算结果，依赖未变就复用上次值，不重新执行工厂函数
- 两个核心用途：
  1. 跳过昂贵计算（排序、过滤、复杂派生数据）
  2. 稳定引用，配合 `React.memo` / `useEffect` 依赖

```jsx
const sorted = useMemo(() => bigList.sort(cmp), [bigList]);
const config = useMemo(() => ({ id, type }), [id, type]);
```

#### 典型组合用法

```jsx
const Child = React.memo(({ data, onClick }) => { /*...*/ });

function Parent() {
  const data = useMemo(() => ({ id: 1 }), []);
  const onClick = useCallback(() => doSomething(), []);
  return <Child data={data} onClick={onClick} />;
}
```

> `useCallback(fn, deps)` 等价于 `useMemo(() => fn, deps)`，是给函数用的语法糖。

### 易错点

- **memo 失效**：父组件传字面量对象 / 数组 / 内联函数 / JSX children，每次都是新引用，浅比较必然失败。
- **滥用 memo**：组件本身渲染极轻，浅比较开销可能比直接渲染还大。
- **依赖数组写错**：`useMemo` 漏依赖会拿到陈旧闭包，多依赖又会频繁失效。
- 把 `useMemo` 当成「保证函数只跑一次」用 → React 不保证，可能因为内存压力丢弃缓存重新计算。
- `React.memo` 只浅比较，嵌套对象变化检测不到，需要自定义 `areEqual` 或保证引用不变。

### 可能的追问

- `React.memo` 和类组件的 `PureComponent` 有什么区别？→ 本质相同，都是 props/state 浅比较
- `useMemo` 和 `useCallback` 的关系？→ 后者是前者的特化版
- 什么时候不该用 memo？→ 浅渲染、props 频繁变化、计算极便宜
- `useMemo` 能保证值「永远不变」吗？→ 不能，React 可能丢缓存；想绝对稳定用 `useRef`
- memo 的子组件 props 包含 `children` 怎么办？→ JSX children 每次都是新对象，需要把 children 也用 `useMemo` 缓存或换思路

### 知识延伸

- `useCallback` 的实现与使用场景
- `useRef` 用来持有「不参与渲染的可变值」的语义
- React 19 的 React Compiler（自动 memo 化），可能让手写 memo 变得不再必要
- 性能优化方法论：先 Profiler 定位 → 拆分组件减少渲染范围 → 必要时 memo
- 列表渲染优化：稳定 `key`、虚拟列表（react-window / react-virtualized）

## 综合记忆

> 渲染触发的本质：**state 变化 → 自己渲染**；**父亲渲染 → 自己默认跟着渲染**；**Context / 外部 store 变化 → 订阅者渲染**。  
> 优化的本质：**`React.memo` 拦截组件渲染**，**`useMemo` 拦截值的计算**，二者都依赖**引用稳定性**才能生效，所以工程上常和 `useCallback` 一起出现。
