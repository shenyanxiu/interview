## 第 1 题：百万级数据 ECharts 渲染优化

**主题**：大数据渲染 / 性能优化  
**难度**：高级

### 问题

基于 antd、echart 渲染百万级数据 CPU 频点图做到毫秒级渲染：
1. 百万级数据从后端拿到前端，数据结构和传输上有没有做优化？
2. ECharts 渲染百万级点位，具体做了哪些优化？
3. 渲染完之后用户交互（缩放、拖动、tooltip）在百万级数据下怎么保证流畅？

### 参考答案

**数据传输层**：
- 后端先做一层降采样（按时间窗口聚合 min/max/avg），前端拿到几万级别即可
- 如果必须前端拿全量，传输用二进制协议（Protobuf、ArrayBuffer）替代 JSON
- 数据结构用 `[[t1,v1],[t2,v2]]` 二维数组而非对象数组，省内存且 ECharts 原生支持

**前端处理层**：
- 原始数据用 `Object.freeze` 或 `markRaw` 隔离响应式（Vue3 的 reactive 对大数组开销极大）
- 双重采样：原始数据 → 手动 min/max 分桶（保留极值）→ LTTB 视觉采样 → 渲染
- 重采样时从「原始数据」算，不能从已采样数据再采（精度逐级丢失）
- 缩放交互绑定 dataZoom 的 throttle（默认 100ms），避免高频触发重采样

**渲染层**：
- `large: true` 启用大数据渲染模式
- `progressive / progressiveThreshold`：渐进式渲染，避免主线程一次性阻塞
- renderer 选 `'canvas'` 不用 `'svg'`
- `animation: false` 关闭动画
- `useDirtyRect: true` 脏矩形渲染

**进阶**：
- 重采样丢到 Web Worker，主线程只负责接收结果
- 极端场景用 OffscreenCanvas + Worker 渲染

### 易错点

- markRaw 拼写（不是 markrow）
- 从已采样数据再采样会导致精度逐级丢失
- 忘记关闭动画是大数据场景最常见的性能黑洞
- 每次交互重采样不加防抖/节流会导致卡顿

### 知识延伸

- ECharts 5 的 dataset + transform 管道
- Web Worker 与 SharedArrayBuffer 在大数据场景的应用
- Canvas 离屏渲染（OffscreenCanvas）

## 第 2 题：Vue3 响应式 markRaw / shallowRef 选型

**主题**：Vue3 响应式原理  
**难度**：中级

### 问题

1. Vue3 的 Proxy 相比 Vue2 的 Object.defineProperty，为什么处理大数组性能反而可能更差？
2. markRaw、shallowRef、shallowReactive、Object.freeze 在「避免大数据被深度响应式」场景下有什么区别？
3. 如果一个对象已经被 reactive 包过了，再对它 markRaw 还有用吗？

### 参考答案

**Proxy 对大数组更慢的原因**：
- Vue2 数组只重写 7 个变更方法，普通 `arr[i]` 读取不触发拦截
- Vue3 Proxy 全方位拦截，每次 get/set 都走陷阱函数。百万级遍历 = 百万次 Proxy.get + 依赖收集判断
- reactive 深度响应，数组里的对象元素都会被递归 Proxy 化
- V8 对原生数组的优化（packed elements、SMI）套上 Proxy 后全部失效

**四个 API 对比**：

| API | 作用 | 是否可追踪 | 能否修改 |
|-----|------|-----------|---------|
| reactive | 深度代理 | 是 | 是 |
| shallowReactive | 只代理第一层 | 第一层是 | 是 |
| shallowRef | 只追踪 .value 整体替换 | 整体替换触发 | 是 |
| markRaw | 永久标记不被代理 | 否 | 是 |
| Object.freeze | JS 原生冻结 | 否（Vue 跳过） | 否 |

**最佳组合**（百万级数据每次交互重采样整体替换）：
```javascript
const chartData = shallowRef(markRaw(sampledData))
```

**reactive 之后再 markRaw 无效**：
- markRaw 给对象加 `__v_skip` 标记，reactive 创建 Proxy 前检查
- 已经是 Proxy 了，标记加在 Proxy 上无效
- 正确做法：markRaw 必须在 reactive 之前

### 易错点

- 认为 Proxy 一定比 defineProperty 快（大数据遍历场景恰恰相反）
- reactive 之后再 markRaw 以为有效（实际无效）
- shallowRef 内部修改不触发更新（必须整体替换 .value）

### 知识延伸

- Vue3 的 toRaw() 获取原始对象
- Vue3 响应式的 WeakMap → Map → Set 三层结构
- triggerRef() 手动触发 shallowRef 更新

## 第 3 题：Vue3 依赖收集 track / trigger 机制

**主题**：Vue3 响应式原理  
**难度**：高级

### 问题

1. 从「读 state.count」到「依赖被收集」，Vue3 内部发生了什么？涉及哪些核心数据结构？
2. state.count++ 触发时，怎么找到应该重新执行哪些 effect？
3. 一个 effect 同时读了 count 和 name，依赖关系怎么存储？为什么不能用 Map<key, effect>？

### 参考答案

**核心数据结构**：
```
WeakMap<target, Map<key, Set<effect>>>
```
- WeakMap：以原始对象为 key，对象回收时自动清理
- Map：每个属性独立维护依赖
- Set：同一属性可被多个 effect 依赖，需去重

**依赖收集流程**：
1. Proxy get 陷阱触发 → 调用 `track(target, 'count')`
2. 检查全局 `activeEffect`（当前正在执行的 effect）
3. WeakMap 找 target 的 Map（depsMap）
4. depsMap 找 'count' 的 Set（dep）
5. 把 activeEffect 加进 Set
6. 反向记录：activeEffect.deps 存 Set 引用（用于清理）

**触发更新流程**：
1. set 陷阱 → `trigger(target, key)`
2. 从 WeakMap 拿 depsMap → 拿 key 的 dep（Set）
3. 遍历 Set 里的 effect，通过 scheduler 推入微任务队列批量执行

**为什么不能用 Map<key, effect>**：
- 一个属性可被多个 effect 依赖（组件渲染 + watchEffect + computed）
- 需要 Set 解决「多对多」+ 去重
- effect 有条件分支时需要 cleanup 再重新收集（避免分支没走到的属性误触发）

### 伪代码

```javascript
let activeEffect = null

function effect(fn) {
  const _effect = () => {
    cleanup(_effect)
    activeEffect = _effect
    fn()
    activeEffect = null
  }
  _effect.deps = []
  _effect()
}

function track(target, key) {
  if (!activeEffect) return
  let depsMap = targetMap.get(target)
  if (!depsMap) targetMap.set(target, depsMap = new Map())
  let dep = depsMap.get(key)
  if (!dep) depsMap.set(key, dep = new Set())
  dep.add(activeEffect)
  activeEffect.deps.push(dep)
}

function trigger(target, key) {
  const dep = targetMap.get(target)?.get(key)
  if (!dep) return
  new Set(dep).forEach(effect => effect())
}
```

### 易错点

- 把 Vue3 的 dep 说成「数组」（是 Set）
- 把 Vue2 的 Dep.notify 和 Vue3 混淆
- 忘记 activeEffect 这个全局变量是依赖收集的关键
- 不知道 cleanup 机制（条件分支场景会产生过期依赖）

### 知识延伸

- Vue3.4 的信号（Signal）优化方向
- effect scope（effectScope）用于批量管理 effect 生命周期
- computed 本质是一个 lazy effect

## 第 4 题：LSP 核心思想与自动补全链路

**主题**：IDE 开发 / LSP  
**难度**：中级

### 问题

1. LSP 的核心思想是什么？解决了什么问题？
2. 自动补全从用户敲字符到列表弹出，完整链路是什么？
3. 高性能模式性能提升约 20%，具体优化了哪些点？

### 参考答案

**LSP 核心 — M × N 问题**：
- M 门语言 × N 个编辑器 = M×N 套实现
- LSP 定义标准协议（JSON-RPC 2.0），解耦语言能力提供方（Server）和编辑器（Client）
- M × N → M + N

**自动补全完整链路**：
1. 用户敲字符 → Client 发 `textDocument/didChange` 通知
2. 触发补全 → Client 发 `textDocument/completion` 请求（uri + position + context）
3. Server 解析文档 → 定位 AST 节点 → 计算候选项
4. 返回 `CompletionItem[]`（label、kind、insertText、documentation）
5. 用户选中 → 可能发 `completionItem/resolve` 懒加载详情
6. 确认 → 插入文本 + 可能的 textEdits

**ux 插件的实现层**（embedded language service）：
- ux 文件虚拟转换为 jsx
- 喂给 TypeScript Language Service 复用补全/类型检查/跳转
- 结果做位置反向映射（source map 思路）回原始 ux 文件
- 本质同 Volar 处理 .vue 单文件的方式

**高性能模式优化点**：
- swc 替代 tsc 做语法解析和转译
- AST 缓存 + 增量解析
- 虚拟 jsx 文档生成结果缓存
- 高频请求防抖
- completion/hover 和 diagnostics 分优先级
- 重活丢 worker/子进程

### 易错点

- 只说「分离」不说 M×N 问题
- 混淆 LSP 链路和具体实现层
- 不知道 completionItem/resolve 的懒加载优化

### 知识延伸

- LSP 的 capabilities 协商机制
- 增量文档同步（incremental sync）vs 全量同步
- Volar 的 virtual document 架构

## 第 5 题：VSCode 插件进程模型与调试变量获取

**主题**：IDE 开发 / VSCode 架构  
**难度**：中级

### 问题

1. VSCode 插件运行在哪个进程？为什么这么设计？
2. 调试模式下动态获取变量值并预览怎么实现？
3. 插件里同步代码很慢会发生什么？怎么避免？

### 参考答案

**进程模型**：
- Main Process：Node.js，窗口管理、生命周期
- Renderer Process：Chromium，Monaco 编辑器、UI
- **Extension Host**：独立 Node.js 进程，所有插件运行于此

设计原因：
- 安全隔离：插件崩溃不影响编辑器
- 性能隔离：插件重活不阻塞 UI 渲染
- API 收敛：插件不能直接操作 DOM，所有调用经 IPC

**调试变量获取**（DAP 协议）：
1. `vscode.debug.activeDebugSession` 获取调试会话
2. 监听 `onDidStartDebugSession` / `onDidTerminateDebugSession`
3. `session.customRequest('evaluate', { expression, frameId, context: 'hover' })`
4. Debug Adapter 在调试上下文求值返回结果
5. 构造 `MarkdownString` 嵌入图片返回 hover

**同步代码阻塞的影响**：
- Extension Host 是单线程，所有插件共享事件循环
- 一个插件卡主线程 → 所有插件命令/事件回调排队 → 语言功能失去响应
- 用户感知「编辑器假死」

**避免方式**：
- 异步化：`fs.promises`、`setImmediate` 切片
- Worker Threads：CPU 密集任务丢 worker
- 子进程：重活开 child_process
- 防抖节流：高频事件 debounce
- 用户反馈：`vscode.window.withProgress` 显示进度

### 易错点

- 不知道 Extension Host 这个名字
- 以为插件阻塞只影响自己（实际影响所有插件）
- 不知道 DAP（Debug Adapter Protocol）的存在

### 知识延伸

- VSCode 的 webview 是独立渲染进程（iframe + postMessage）
- Extension Host 的 unresponsive 检测机制
- LSP server 为什么要单独起进程（和 Extension Host 隔离）

## 第 6 题：V8 JIT / 隐藏类 / 数组优化

**主题**：V8 / JavaScript 引擎  
**难度**：高级

### 问题

1. 什么是 JIT？为什么 V8 用 JIT 而不是解释执行或 AOT？
2. 隐藏类（Hidden Class）是什么？为什么对象属性添加顺序要一致？
3. `const arr = []; for(let i=0;i<1e6;i++) arr[i]=i;` 有什么性能问题？

### 参考答案

**JIT vs 解释 vs AOT**：

| 方式 | 时机 | 优点 | 缺点 |
|------|------|------|------|
| 解释执行 | 逐行翻译 | 启动快 | 运行慢 |
| AOT | 运行前全编译 | 运行极快 | 启动慢、JS 动态类型无法完整静态编译 |
| JIT | 运行时按需编译热点 | 兼顾启动和运行 | 实现复杂 |

V8 选 JIT 的原因：
- JS 动态类型，AOT 无法做类型特化优化
- JIT 运行时收集类型信息（type feedback），针对性编译
- 类型假设失败可 deopt 回退，AOT 做不到

V8 流程：源码 → AST → Ignition（字节码）→ 热点 → TurboFan（机器码）→ deopt 回退

**隐藏类**：
- V8 给每个对象分配隐藏类（Map），描述对象形状（属性 + 偏移量）
- 属性添加形成转换链（transition chain）
- 相同顺序添加属性 → 共享隐藏类 → Inline Cache 命中 → 接近 C struct 速度
- 不同顺序 → 不同隐藏类 → IC 失效 → 退化为多态查找，慢 5-10 倍

**数组 Elements Kind**：
- 空数组循环赋值 → 多次扩容（约 40+ 次内存拷贝）
- 更快写法：`new Array(1e6)` 预分配 或 `new Int32Array(1e6)`
- TypedArray 优势：固定长度/类型、连续内存、CPU cache 友好

### 易错点

- 认为 JIT 和 AOT 是对立的（实际 V8 混合使用）
- 不知道 delete 属性会让对象退化为字典模式
- `new Array(1e6)` 创建的是 HOLEY 数组（有空洞），比 PACKED 慢

### 知识延伸

- V8 的 Inline Cache（IC）机制
- Monomorphic / Polymorphic / Megamorphic 状态
- TurboFan 的 Sea of Nodes IR

## 第 7 题：MCP 协议与 Agent 性能优化流程

**主题**：AI 工程化  
**难度**：中级

### 问题

1. MCP 是什么？解决什么问题？和直接写 prompt 调工具有什么区别？
2. 用 chrome devtool MCP 做性能优化的流程？
3. vibe coding 生成的代码质量怎么保证？

### 参考答案

**MCP 核心**：
- 类比 LSP：LSP 解耦语言和编辑器，MCP 解耦模型和工具
- M 个工具 × N 个 AI 应用 → M + N
- 基于 JSON-RPC 2.0，三个角色：Host、Client、Server
- Server 暴露 tools / resources / prompts
- 动态发现：运行时查询可用工具

**和 Function Calling 的区别**：
- Function Calling：模型层面能力（输出结构化调用意图）
- MCP：应用层面协议（工具怎么暴露、发现、调用）
- 互补关系：模型通过 FC 决定「调什么」，MCP 负责「怎么调到」

**性能优化流程**：
1. Skill 定义工作流（目标 + 可用工具 + 步骤）
2. Agent 调用 MCP 的 lighthouse_audit 获取性能数据
3. 分析瓶颈（LCP 慢、资源阻塞等）
4. 根据瓶颈决定优化策略并修改代码
5. 再次调用 MCP 验证效果
6. 人工 review 最终改动

**代码质量保证三层**：
- 行为正确性：AI 生成 Playwright E2E 测试
- 代码质量：ESLint + TypeScript 严格模式 + hook 自动跑检查
- 性能保证：benchmark 测试 + Lighthouse CI 持续监控

### 易错点

- 混淆 MCP 和 Function Calling（层次不同）
- 认为 Agent 优化性能是「一步到位」（实际是循环验证）
- vibe coding 只靠 E2E 测试不够（还需要 lint + 性能守护）

### 知识延伸

- MCP 的 stdio / HTTP SSE 通信方式
- Tool Use 的 schema 定义规范
- AI 代码生成的 adversarial review 策略

## 第 8 题：单 Agent vs 多 Agent / ReAct 循环

**主题**：AI Agent 架构  
**难度**：中级

### 问题

1. 单 Agent 和多 Agent 架构的区别？各自适合什么场景？
2. ReAct 循环的基本流程是什么？为什么不能一步到位？
3. Agent 陷入死循环或错误决策怎么处理？

### 参考答案

**单 Agent vs 多 Agent**：

| 维度 | 单 Agent | 多 Agent |
|------|---------|---------|
| 结构 | 一个模型 + 一套工具 + 一个上下文 | 多个独立角色/上下文/工具集 |
| 适合 | 任务链路短、工具少 | 复杂任务需要不同专业能力 |
| 优点 | 简单、延迟低 | 专业分工、上下文不爆 |
| 缺点 | 上下文有限、复杂任务易跑偏 | 协调成本高、延迟大 |

多 Agent 协作方式：
- 顺序编排（Pipeline）：A → B → C
- 层级编排（Orchestrator）：主 Agent 分配子任务
- 协商式（Debate）：多 Agent 讨论达成共识

**ReAct 循环**：
```
Thought → Action → Observation → Thought → Action → ... → Final Answer
```

为什么不能一步到位：
- LLM 不是全知的，必须先观察再决策
- 复杂任务有依赖链（后步依赖前步结果）
- 环境是动态的，修改后效果未知需验证
- 单步输出容易幻觉，循环验证能纠错

**Agent 失败模式与应对**：
- 死循环 → 设置 max_iterations + 检测重复 action
- 幻觉工具调用 → 严格 tool schema 校验
- 上下文丢失 → 关键信息放 system prompt + memory 机制
- 过度自信 → 强制验证步骤
- 错误扩散 → 每步 sanity check + human-in-the-loop

### 易错点

- 认为多 Agent 一定比单 Agent 好（简单任务多 Agent 反而慢）
- 不知道 ReAct 是 Agent 的基础范式
- 忽略 Agent 的失败处理机制

### 知识延伸

- Plan-and-Execute 模式（先规划再执行）
- Tool Use vs RAG 的选择
- Agent Memory（短期/长期记忆）架构

## 第 9 题：SSE 流式输出与 Markdown 实时渲染

**主题**：前端流式渲染  
**难度**：中级

### 问题

1. 流式输出在前端怎么实现？用什么协议/API？
2. AI 回复包含 Markdown + 代码块，实时渲染有什么难点？
3. 用户点「停止生成」，前端和后端分别做什么？

### 参考答案

**SSE 实现**：
```javascript
const response = await fetch('/api/chat', {
  method: 'POST',
  body: JSON.stringify({ message }),
  signal: abortController.signal
});

const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  const chunk = decoder.decode(value, { stream: true });
  // 解析 SSE 格式，增量更新 UI
}
```

**SSE vs WebSocket**：

| 维度 | SSE | WebSocket |
|------|-----|-----------|
| 方向 | 单向（服务端→客户端） | 双向 |
| 协议 | HTTP | 独立协议 ws:// |
| 重连 | 浏览器自动 | 需手动 |
| 适用 | AI 流式输出 | 实时聊天/协同编辑 |

AI 选 SSE 原因：天然单向、更轻量、兼容负载均衡和 CDN。

**流式 Markdown 渲染难点**：
- 代码块未闭合时解析器报错 → 临时补 ``` 让解析正确
- 频繁 re-render → requestAnimationFrame / throttle 控制频率
- 维护完整 buffer，只重解析最后一个不确定块
- React 用 useDeferredValue 或批量 setState
- 长对话用虚拟滚动

**停止生成**：

前端：
- `AbortController.abort()` 中断 fetch
- `reader.cancel()` 关闭流
- UI 状态切换，保留已生成内容
- 清理定时器/动画副作用

后端：
- 检测客户端断开连接
- 停止调用 LLM API（也要 abort）
- 释放推理资源
- 可选：记录已生成部分到数据库

### 易错点

- 混淆 EventSource（只支持 GET）和 fetch + ReadableStream
- 不处理未闭合代码块导致渲染错误
- 停止生成后丢弃已生成内容（应保留）
- 忘记后端也要停止（否则浪费算力）

### 知识延伸

- HTTP/2 Server Push vs SSE
- ReadableStream 的 TransformStream 管道处理
- React 18 的 Streaming SSR 与流式渲染的关系
