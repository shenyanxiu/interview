# Kafka 消费者

## 1. Consumer Group 是什么？有什么作用？

**主题**：消费者组  
**难度**：初级

### 参考答案

Consumer Group 是 Kafka 消费侧的核心组织单位：

- **同一消费组内**：一个分区只能被一个消费者消费（保证顺序、不重复）
- **不同消费组之间**：互不影响，可重复消费同一份数据（实现广播）
- 消费者数 > 分区数时，多余的消费者**空闲**

### 示意图

```
Topic A (4 个分区)
┌────┐ ┌────┐ ┌────┐ ┌────┐
│ P0 │ │ P1 │ │ P2 │ │ P3 │
└─┬──┘ └─┬──┘ └─┬──┘ └─┬──┘
  │      │      │      │
  ▼      ▼      ▼      ▼
┌────────────────────────┐    ┌──────────────────┐
│  Consumer Group 1      │    │ Consumer Group 2 │
│  C1   C2   C3   C4     │    │       C1         │
│  P0   P1   P2   P3     │    │   P0,P1,P2,P3    │
└────────────────────────┘    └──────────────────┘
   每人 1 分区                  独占消费所有分区
```

### 业务场景

```
order-created 消息
    │
    ├──► 消费组 A：订单服务（处理订单逻辑）
    ├──► 消费组 B：通知服务（发短信）
    └──► 消费组 C：数据分析服务（写数据仓库）
```

每个组独立消费一份完整数据，组内成员分工。

- 消费者数应该和分区数保持什么关系？

## 2. 一个分区可以被同组多个消费者消费吗？

**主题**：消费规则  
**难度**：初级

### 参考答案

**不可以。** 同一消费组内，一个分区**只能**被一个消费者消费。

### 原因

- 保证**消息顺序**：同分区消息有序，多消费者会乱序
- 避免**重复消费**：offset 只有一份，多消费者会冲突

### 推论

- 消费者数 ≤ 分区数（多余的消费者会空闲）
- 想增加消费并行度 → 增加分区数
- 分区数是消费侧扩容的上限

## 3. Rebalance（再均衡）什么时候触发？有什么影响？

**主题**：消费者协调  
**难度**：中级

### 参考答案

#### 触发时机

1. **消费者加入或退出**消费组
2. **订阅 Topic 数量变化**（动态订阅）
3. **Topic 的分区数变化**（增加分区）
4. 消费者**长时间未发送心跳**被踢出

#### Rebalance 的影响

- **消费暂停（STW）**：rebalance 期间消费者无法工作
- **重复消费风险**：未提交 offset 的消息会被重新消费
- **频繁 rebalance** 会严重影响性能

#### 心跳相关参数

| 参数 | 默认值 | 说明 |
|---|---|---|
| `heartbeat.interval.ms` | 3s | 心跳发送间隔 |
| `session.timeout.ms` | 10s | 会话超时（无心跳判死时长）|
| `max.poll.interval.ms` | 5min | 两次 poll 最大间隔（处理超时判死）|

### 怎么避免频繁 Rebalance？

1. 消费速度跟得上，避免 `max.poll.interval.ms` 超时
2. 合理设置 `max.poll.records`，避免一次拉太多处理不完
3. 心跳和 session 超时配置合理（一般 `heartbeat = session/3`）
4. 使用 CooperativeSticky 策略减少 STW

## 4. Rebalance 的分区分配策略有哪些？

**主题**：分区分配  
**难度**：中级

### 参考答案

| 策略 | 特点 | 缺点 |
|---|---|---|
| **Range** | 按 Topic 范围分 | 多 Topic 时不均匀 |
| **RoundRobin** | 全部分区轮询分 | 订阅不一致时有问题 |
| **Sticky** | 粘性分配，尽量保留原分配 | 复杂度高 |
| **CooperativeSticky** | 合作式粘性，**渐进式** rebalance | 推荐 |

### Range（默认）

按 Topic 算，分区数 / 消费者数，前几个消费者多拿一个：

```
T1 有 7 个分区，3 个消费者
C1: P0, P1, P2  (3 个)
C2: P3, P4      (2 个)
C3: P5, P6      (2 个)
```

问题：多 Topic 时，前几个消费者每个 Topic 都多拿一个，整体不均衡。

### RoundRobin

把所有 Topic 的所有分区合起来轮询：

```
T1: P0~P3, T2: P0~P3
C1: T1-P0, T2-P0, T1-P2, T2-P2
C2: T1-P1, T2-P1, T1-P3, T2-P3
```

### Sticky

每次 rebalance 尽量让消费者保留之前的分区，减少分区迁移：

```
原分配: C1=[P0,P1,P2], C2=[P3,P4,P5]
C2 挂了
重分配: C1=[P0,P1,P2,P3], C3=[P4,P5]   ← P0,P1,P2 不变
```

### CooperativeSticky（推荐）

Sticky 的升级版，**渐进式 rebalance**：

- 不需要一次性所有消费者全停（不再 STW）
- 只让需要变动的消费者短暂暂停
- 大集群下 rebalance 时间从几十秒降到秒级

### 知识延伸

- Kafka 2.4+ 引入 CooperativeSticky
- 协议层引入了 `revoke` / `assign` 的两阶段流程

## 5. offset 是什么？有哪些提交方式？

**主题**：消费进度  
**难度**：中级

### 参考答案

**offset** 是消息在分区内的唯一编号，消费者用它记录"我读到第几条了"。

消费进度存储在 Kafka 内部 Topic `__consumer_offsets` 中，按 `<group, topic, partition>` 维度记录。

### 三种提交方式

#### 1. 自动提交（默认）

```properties
enable.auto.commit=true
auto.commit.interval.ms=5000
```

- 后台定时（默认 5s）提交
- 简单但不可控，可能重复或丢失

#### 2. 手动同步提交

```java
consumer.commitSync();
```

- 阻塞直到提交成功
- 失败会自动重试
- 性能较差

#### 3. 手动异步提交

```java
consumer.commitAsync((offsets, exception) -> { ... });
```

- 不阻塞
- 失败**不重试**（避免提交位移倒退）
- 需自己处理失败回调

### 最佳实践

```java
try {
    while (true) {
        ConsumerRecords records = consumer.poll(timeout);
        process(records);
        consumer.commitAsync();  // 平时用异步
    }
} finally {
    try {
        consumer.commitSync();   // 关闭前最后一次用同步
    } finally {
        consumer.close();
    }
}
```

### 追问链

- offset 提交早了会怎样？（消息丢失）
- offset 提交晚了会怎样？（重复消费）
- `__consumer_offsets` 是什么？（Kafka 内部 Topic，默认 50 分区）

## 6. 消费者挂了，正在处理的消息会丢吗？会被其他消费者处理吗？

**主题**：故障恢复  
**难度**：中级

### 参考答案

**结论：消息不会丢，会重新分配给同组其他消费者，但可能重复消费。**

### 关键原理

Kafka 不是"推"消息，而是消费者主动**拉取并记录进度**。消息一直存在 Broker 磁盘，消费者只是带着 offset 去拉。

### 故障处理流程

#### 第 1 步：检测消费者挂了

消费者每 `heartbeat.interval.ms`（默认 3s）发心跳。超过 `session.timeout.ms`（默认 10s）没心跳，Coordinator 判定它挂了。

#### 第 2 步：触发 Rebalance

Coordinator 把它负责的分区**重新分配**给同组其他消费者。

```
原分配:                    Rebalance 后:
P0 → A                     P0 → B  ✨
P1 → A     A 挂了 💥       P1 → C  ✨
P2 → B     ────────►       P2 → B
P3 → C                     P3 → C
```

#### 第 3 步：新消费者从最后提交的 offset 开始读

- 如果 A 已提交 offset=100 → B 从 101 开始（不重不丢）
- 如果 A 处理到 105 但只提交了 100 → B 从 101 开始（101~105 重复消费）

### 三种 offset 策略下的表现

```js
// ❌ 策略 1：先提交后处理 → 可能丢失
const msg = poll()
commit(msg.offset)
process(msg)         // 这里挂了，消息丢了

// ✅ 策略 2：先处理后提交 → 可能重复（推荐）
const msg = poll()
process(msg)         // 处理时挂了
commit(msg.offset)   // 没执行，下次会重读

// ⚠️ 策略 3：自动提交 → 不可控
```

### Kafka 默认承诺：至少一次（At Least Once）

- 消息**绝对不会丢**（Broker 持久化 + ISR 副本）
- 消费侧**可能重复**（rebalance + 未提交 offset）
- **业务必须做幂等**

### 业务幂等的常见做法

```js
async function processMessage(msg) {
  const exists = await redis.exists(`processed:${msg.id}`)
  if (exists) return  // 已处理过

  await businessLogic(msg.data)

  await redis.set(`processed:${msg.id}`, 1, 'EX', 86400)
}
```

或者：

- 数据库唯一索引去重
- 状态机幂等更新（订单 pending → paid，重复请求无效）
