# Kafka 生产者

## 1. Producer 发送消息的完整流程是怎样的？

**主题**：Producer 原理  
**难度**：中级

### 参考答案

```
应用调用 send()
    │
    ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│  拦截器   │ ─► │ 序列化器  │ ─► │  分区器   │
└──────────┘    └──────────┘    └──────────┘
                                     │
                                     ▼
                          ┌──────────────────────┐
                          │ RecordAccumulator     │
                          │  (双端队列缓冲池)      │
                          │  按分区分组成 Batch    │
                          └──────────┬───────────┘
                                     │
                                     ▼
                          ┌──────────────────────┐
                          │   Sender 线程         │
                          │  批量拉取 → 网络发送   │
                          └──────────┬───────────┘
                                     │
                                     ▼
                                  Broker
```

### 关键步骤说明

1. **拦截器**：可选，用于消息发送前的统一处理（埋点、加签等）
2. **序列化器**：把对象序列化成字节数组（key/value 各自序列化）
3. **分区器**：决定消息发到哪个分区
4. **RecordAccumulator**：消息按分区聚合成 Batch，提升吞吐
5. **Sender**：独立 IO 线程，从缓冲区拉取 Batch 发往 Broker

### 关键参数

| 参数 | 默认值 | 说明 |
|---|---|---|
| `batch.size` | 16KB | Batch 大小，攒够就发 |
| `linger.ms` | 0 | 等待时间，超时也发 |
| `buffer.memory` | 32MB | 缓冲区总大小 |
| `max.in.flight.requests.per.connection` | 5 | 单连接未确认请求数 |
| `compression.type` | none | 压缩类型 |

### 追问链

- 为什么要有 RecordAccumulator？（批量提升吞吐）
- Sender 是同步还是异步？（独立线程异步发送）
- Producer 是线程安全的吗？（是，可以多线程共享）

## 2. Producer 的分区策略有哪些？

**主题**：Producer 分区  
**难度**：初级

### 参考答案

按优先级：

1. **指定 partition**：直接发送到指定分区
2. **指定 key**：`hash(key) % 分区数`，相同 key 一定到相同分区
3. **都不指定**：**粘性分区（Sticky Partitioning）**，一批数据发同一分区，攒满或超时再换分区

### Sticky Partitioner 的优势

旧版本是轮询，会导致每个分区都只有零散消息，Batch 凑不满。粘性分区让一个 Batch 内消息集中到一个分区，提升了批量发送效率。

### 应用场景

- **顺序保证**：相同业务 key（如订单 ID）发同一分区 → 同一订单的消息有序
- **负载均衡**：不指定 key，依赖粘性分区均衡分布

### 自定义分区器

实现 `Partitioner` 接口，重写 `partition()` 方法。

### 追问链

- 想要全局有序怎么办？（Topic 设 1 个分区）
- 想要业务有序怎么办？（业务 key 作为消息 key）

## 3. acks 参数的三种取值有什么区别？

**主题**：消息可靠性  
**难度**：中级

### 参考答案

`acks` 控制 Producer 认为消息发送成功的条件：

| acks | 含义 | 可靠性 | 性能 | 场景 |
|---|---|---|---|---|
| `0` | 不等确认，发出去就行 | 最低，可能丢 | 最高 | 日志类容忍丢失 |
| `1` | Leader 写入即确认 | 中等，Leader 挂会丢 | 中 | 普通场景默认 |
| `-1` / `all` | ISR 全部写入才确认 | 最高 | 最低 | 金融、订单 |

### acks=-1 + min.insync.replicas

光设 `acks=-1` 不够，还要配合 `min.insync.replicas`：

```
replication.factor=3
min.insync.replicas=2
acks=-1
```

效果：ISR 至少 2 个副本同步成功才算写入成功，否则 Producer 收到异常。

### 易错点

- `acks=-1` 是等 ISR 全部，不是等所有副本
- 如果 ISR 缩到只剩 Leader，`acks=-1` 就等于 `acks=1`
- `min.insync.replicas` 是 Broker / Topic 级别配置

## 4. Producer 的幂等性和事务是什么？

**主题**：Exactly-Once  
**难度**：高级

### 参考答案

#### 幂等性

**作用**：保证 Producer 重试不会导致消息重复。

**开启方式**：

```properties
enable.idempotence=true
```

**实现原理**：

- 每个 Producer 启动时分配唯一 **PID**（Producer ID）
- 每条消息带 **SequenceNumber**（按分区递增）
- Broker 维护 `<PID, Partition>` 的最新 SequenceNumber
- 重复或乱序消息直接拒绝

**局限性**：

- 只保证**单分区、单会话**内不重复
- Producer 重启后 PID 会变，无法跨会话去重

#### 事务

**作用**：跨分区、跨 Topic 的原子写入，实现 **Exactly-Once 语义**。

**典型场景**：消费 - 处理 - 生产模式（流处理）

```java
producer.initTransactions();
try {
    producer.beginTransaction();
    producer.send(record1);
    producer.send(record2);
    // 把消费 offset 也纳入事务
    producer.sendOffsetsToTransaction(offsets, consumerGroupId);
    producer.commitTransaction();
} catch (Exception e) {
    producer.abortTransaction();
}
```

**关键配置**：

```properties
transactional.id=my-tx-1   # 跨会话唯一标识
enable.idempotence=true    # 事务必须开启幂等
```

#### Exactly-Once 三件套

要实现端到端的 Exactly-Once：

1. Producer 开启幂等 + 事务
2. Consumer 设置 `isolation.level=read_committed`（只读已提交事务的消息）
3. 业务处理与 offset 提交在同一事务内

### 追问链

- 幂等性能跨分区吗？（不能）
- 事务是怎么实现的？（Transaction Coordinator + Transaction Log）
- 为什么 transactional.id 要跨会话保持？（用来 fence 旧 Producer）
