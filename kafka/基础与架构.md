# Kafka 基础与架构

## 1. Kafka 是什么？核心特性是什么？

**主题**：Kafka 基础  
**难度**：初级

### 参考答案

Kafka 是一个分布式、高吞吐、可持久化的消息队列（流处理平台），由 LinkedIn 开源，现属于 Apache。

**核心特性：**

- **高吞吐**：单机可达百万级 TPS，得益于顺序写磁盘、零拷贝、批量发送
- **持久化**：消息存储在磁盘，支持配置保留时间（默认 7 天）
- **分布式**：天然支持水平扩展和高可用
- **多消费者**：同一条消息可被多个消费者组消费
- **顺序性**：单分区内消息有序

### 前端视角类比

Kafka ≈ 后端工业版的 EventBus / 发布订阅系统：

| 前端 EventBus | Kafka |
|---|---|
| 内存里 | 磁盘上（消息能保留几天）|
| 一个浏览器内 | 跨多台服务器 |
| 进程挂了消息就没了 | 进程挂了消息还在 |
| 每秒几千条 | 每秒上百万条 |
| `emit('event', data)` | Producer 发消息到 Topic |
| `on('event', cb)` | Consumer 订阅 Topic |

## 2. Kafka 的核心组件有哪些？

**主题**：Kafka 基础  
**难度**：初级

### 参考答案

| 组件 | 说明 | 前端类比 |
|---|---|---|
| Producer | 消息生产者 | 调 `eventBus.emit` 的人 |
| Consumer | 消息消费者 | 写 `eventBus.on` 的人 |
| Broker | Kafka 服务节点（一台机器一个进程）| 一台后端服务器 |
| Topic | 消息主题（逻辑分类）| 事件名 / 频道名 |
| Partition | 分区，Topic 的物理分片 | 把一个频道拆成多条流水线 |
| Replica | 副本，分为 Leader 和 Follower | 主从备份 |
| Consumer Group | 消费者组 | 一组协同分工的消费者 |
| Offset | 消息在分区内的唯一编号 | B 站视频的进度条 |
| Controller | 集群协调者（从 Broker 选出）| 集群"大脑" |
| ZooKeeper / KRaft | 集群元数据管理 | 配置中心 |

### 易错点

- Topic 是逻辑概念，Partition 才是物理存储单元
- 一个 Broker 上可以有多个 Topic 的多个 Partition
- Controller 是集群中某个 Broker 兼任的角色，不是独立节点

## 3. Kafka 整体架构是什么样的？

**主题**：Kafka 架构  
**难度**：中级

### 参考答案

Kafka 架构分为四层：

#### 架构全景

```
                ZooKeeper / KRaft (元数据管理)
                          ▲
                          │
   Producer ──push──► Kafka Cluster ──pull──► Consumer Group
                  (多个 Broker，每个 Broker 上有多个 Partition)
```

#### 1. 生产者层（Producer）

负责生产消息，内部流程：

```
send() → 拦截器 → 序列化器 → 分区器
       → RecordAccumulator(缓冲区，按分区聚批)
       → Sender 线程批量拉取并发送
```

#### 2. 服务端层（Broker）

每个 Broker 内部包含：

- 网络层（Acceptor + Processor）
- API 层（处理 Produce / Fetch / Metadata 等请求）
- 日志管理（Log Manager）
- 副本管理（Replica Manager）
- 集群中只有一个 Broker 担任 Controller

**Topic / Partition / Replica 物理分布**（Topic A 有 3 分区 3 副本）：

```
Broker 0          Broker 1          Broker 2
P0-Leader         P0-Follower       P0-Follower
P1-Follower       P1-Leader         P1-Follower
P2-Follower       P2-Follower       P2-Leader
```

设计原则：副本和 Leader 都均匀分布，避免单点故障和热点。

#### 3. 存储层（Log Storage）

每个 Partition 一个目录，目录内分多个 Segment（默认 1G 滚动）：

```
topic-A-0/
├── 00000000000000000000.log      数据文件
├── 00000000000000000000.index    位移索引（稀疏）
├── 00000000000000000000.timeindex
└── ...
```

#### 4. 消费者层（Consumer Group）

```
Topic A (4 分区) → Consumer Group (4 个消费者，每人 1 分区)
```

### 一条消息的完整生命周期

1. Producer 调用 send()
2. 序列化 + 分区器选择分区
3. 进入缓冲区聚批
4. Sender 批量发送到 Leader Broker
5. Leader 顺序写入磁盘（先 PageCache）
6. Follower 主动拉取同步
7. ISR 全部同步后更新 HW
8. 根据 acks 返回 ack
9. Consumer 从 Leader 拉取消息
10. 业务处理后提交 offset 到 `__consumer_offsets`
11. 按保留策略最终删除

- 为什么 Kafka 要去掉 ZooKeeper？

## 4. Kafka 为什么这么快？

**主题**：Kafka 性能  
**难度**：高级

### 参考答案

Kafka 高吞吐的核心原因有 6 点：

#### 1. 顺序写磁盘

顺序写速度接近内存随机写，远快于磁盘随机写。Kafka 所有消息追加写入日志文件末尾，没有随机 IO。

#### 2. PageCache（页缓存）

Kafka 不维护进程内缓存，直接依赖操作系统 PageCache：

- 写：先写 PageCache，由 OS 异步刷盘
- 读：优先从 PageCache 读，命中率高时几乎没磁盘 IO

#### 3. 零拷贝（Zero-Copy）

使用 `sendfile` 系统调用，数据从 PageCache 直接到网卡：

```
传统 4 次拷贝:  Disk → Kernel → User → Kernel → NIC
零拷贝 2 次:    Disk → PageCache → NIC
```

避免了内核态 / 用户态切换开销。

#### 4. 批量发送 + 压缩

- Producer 攒一批再发（`batch.size` + `linger.ms`）
- 支持 gzip / snappy / lz4 / zstd 压缩
- 网络传输的是压缩后的批次，Broker 直接存储压缩数据

#### 5. 分区并行

多个 Partition 可以分布在不同 Broker，Producer 和 Consumer 都能并行读写。

#### 6. 稀疏索引

`.log` 数据文件配 `.index` 稀疏索引文件，二分查找快速定位消息（O(logN)）。

### 易错点

- 零拷贝的"零"不是真的零次拷贝，是减少了用户态拷贝
- PageCache 是 OS 行为，不是 Kafka 自己实现的
- 批量发送是 Producer 端的优化，跟 Broker 无关

- Linux `sendfile` 系统调用原理
- mmap 与 sendfile 的区别
- 顺序 IO 和随机 IO 的性能差距实测
