# Kafka 分区与副本

## 1. Topic、Partition、Replica 三者的关系是什么？

**主题**：分区机制  
**难度**：初级

### 参考答案

- 一个 Topic 由多个 Partition 组成（逻辑分类 → 物理分片）
- 每个 Partition 有多个 Replica（副本）
- 副本中只有一个是 Leader，其他是 Follower
- 生产和消费**只与 Leader 交互**，Follower 只做异步同步备份
- Leader 挂了后从 ISR 里选新 Leader

```
Topic A (3 分区, 3 副本)

Broker 0         Broker 1         Broker 2
P0-Leader        P0-Follower      P0-Follower
P1-Follower      P1-Leader        P1-Follower
P2-Follower      P2-Follower      P2-Leader
```

### 设计要点

- **副本均匀分布**在不同 Broker，避免单点故障
- **Leader 分散**在不同 Broker，分散读写压力
- 副本数（replication.factor）≤ Broker 数

### 追问链

- 为什么 Follower 不参与读？（保证一致性，避免读到旧数据）
- 副本数设几个合适？（生产环境一般 3）

## 2. 什么是 ISR、OSR、AR？

**主题**：副本机制  
**难度**：中级

### 参考答案

- **AR（Assigned Replicas）**：所有副本集合
- **ISR（In-Sync Replicas）**：与 Leader 保持同步的副本集合（包括 Leader 本身）
- **OSR（Out-of-Sync Replicas）**：同步落后过多被踢出的副本

**关系：** `AR = ISR + OSR`

### 副本被踢出 ISR 的条件

同步延迟超过 `replica.lag.time.max.ms`（默认 30 秒）。

### 为什么需要 ISR？

- acks=-1 时，只要 ISR 全部同步即可返回 ack，不必等 OSR
- Leader 选举只从 ISR 里选，避免选到落后太多的副本造成数据丢失

### 追问链

- ISR 中的副本数为 0 怎么办？
- 如何监控 ISR 的健康状态？
- `min.insync.replicas` 是什么？

## 3. HW 和 LEO 是什么？

**主题**：副本同步机制  
**难度**：高级

### 参考答案

- **LEO（Log End Offset）**：每个副本最新写入消息的下一个位移（即将写入的位置）
- **HW（High Watermark）**：消费者能看到的最大位移，等于 ISR 中**最小的 LEO**

```
Leader   :  [m1][m2][m3][m4][m5]   LEO=5
Follower1:  [m1][m2][m3][m4]       LEO=4
Follower2:  [m1][m2][m3]           LEO=3
                          ↑
                       HW=3 (取最小)

Consumer 只能消费 offset < 3 的消息
```

### HW 的作用

- **保证已确认消息对消费者可见**：未被 ISR 全部同步的消息，消费者看不到
- **Leader 切换时数据一致性**：新 Leader 上 HW 之前的数据保证存在
- **未提交消息不会丢**：但消费者也不会读到，避免脏读

### 易错点

- HW 不是消息总数，是位移
- HW 由 Leader 维护，每次 Follower 同步后更新
- Follower 自己的 HW 会延迟于 Leader 的 HW

### 知识延伸

- Leader Epoch 解决了 HW 截断的脏数据问题（0.11+ 引入）

## 4. Leader 选举机制是怎样的？

**主题**：高可用  
**难度**：高级

### 参考答案

#### 选举流程

1. 由 **Controller**（Broker 中选出的协调者）负责
2. 监听到 Leader 副本所在 Broker 下线
3. 优先从 **ISR** 中选第一个存活的副本作为新 Leader
4. 通知所有 Broker 更新元数据
5. Producer / Consumer 通过 metadata 请求拿到新 Leader 信息

#### ISR 全挂的特殊情况

看 `unclean.leader.election.enable` 配置：

| 取值 | 行为 | 取舍 |
|---|---|---|
| `false`（推荐）| 等待 ISR 副本恢复 | 牺牲可用性保数据 |
| `true` | 从 OSR 选举新 Leader | 可能丢失数据，但保可用性 |

#### Controller 自身的选举

- ZK 模式：所有 Broker 在 ZK 上抢注 `/controller` 临时节点
- KRaft 模式：通过 Raft 协议选举

### 追问链

- 为什么不让所有 ISR 副本都能写？（一致性问题）
- Controller 挂了怎么办？
- Leader Epoch 是干嘛的？

## 5. 分区数是不是越多越好？

**主题**：分区设计  
**难度**：中级

### 参考答案

**不是。** 分区过多带来的问题：

1. **文件句柄消耗多**：每个 Partition 对应若干文件
2. **Leader 选举耗时长**：Controller 要逐个分区处理
3. **端到端延迟增加**：Follower 同步压力增大
4. **Producer 内存占用增大**：每个分区都有缓冲区
5. **Rebalance 时间变长**：分区数越多，重平衡越慢

### 经验值

- 单个 Broker 上 Partition 总数控制在几千以内
- 单个 Topic 的 Partition 数：根据峰值 TPS / 单分区吞吐 估算
- 一般起步配置：3 ~ 12 个分区

### 怎么估算分区数？

```
分区数 ≈ max(目标吞吐 / 单分区生产吞吐, 目标吞吐 / 单分区消费吞吐)
```

例如：目标 100 MB/s，单分区生产 25 MB/s、消费 50 MB/s，则分区数 ≥ 4。

### 易错点

- 分区数只能增加不能减少
- 分区数 ≥ 消费者数才能让所有消费者都有活干
- 分区数过少会成为吞吐瓶颈
