# 锁与并发同步

## Q1: 锁是什么

### 问题

锁是什么？为什么需要锁？前端开发中有对应的概念吗？

### 参考答案

**锁（Lock）** 是一种同步机制，用来保证同一时刻只有一个执行单元（线程/进程/任务）能访问某段代码或某个共享资源，防止并发冲突。

**为什么需要锁——竞态条件（Race Condition）：**

多个线程同时操作同一份数据时会出现数据丢失。例如 `count++` 实际是"读 → 加 1 → 写回"三步：

- 线程 A 读到 0，还没写回
- 线程 B 也读到 0
- 两个都加 1 写回 1，本应是 2，结果是 1

加锁后保证整段操作不可分割：

```js
lock()
count++   // 同一时刻只有一个执行单元在跑
unlock()
```

**常见锁类型：**

| 类型 | 特点 |
|------|------|
| 互斥锁（Mutex） | 最基础，同一时刻只允许一个线程进入 |
| 读写锁 | 读可并发，写互斥（读多写少场景） |
| 自旋锁 | 拿不到锁就循环重试，不让出 CPU（适合临界区极短） |
| 悲观锁 | 假设一定会冲突，先加锁再操作（如 MySQL `SELECT ... FOR UPDATE`） |
| 乐观锁 | 假设不会冲突，提交时检查版本号（CAS、`version` 字段） |
| 分布式锁 | 跨进程/跨机器加锁（Redis SETNX、Zookeeper） |

**前端的"锁"对应物：**

JavaScript 主线程是单线程的，单条同步逻辑天然不会被打断，所以前端基本不需要写真正的锁。但"抢同一个资源"的场景到处都是，处理思路本质就是锁：

| 前端场景 | 对应的"锁" |
|---------|-----------|
| 防止按钮重复点击 | `loading` 标志（互斥锁） |
| 防抖 / 节流 | 时间窗口锁 |
| 搜索请求竞态 | 版本号 / requestId（乐观锁） |
| Promise 串行队列 | `queue = queue.then(...)`（排队锁） |
| 多标签页协作 | `navigator.locks` API（浏览器原生 Web Locks） |
| Service Worker 更新 | Web Locks API |

**典型例子——按钮防重复点击就是互斥锁：**

```js
let loading = false  // 这就是锁

async function submit() {
  if (loading) return       // 锁被占了，直接返回
  loading = true            // 上锁
  try {
    await createOrder()
  } finally {
    loading = false         // 解锁
  }
}
```

**典型例子——搜索竞态用乐观锁思路：**

```js
let requestId = 0

async function search(keyword) {
  const myId = ++requestId       // 当前请求的"版本"
  const res = await fetch(...)
  if (myId !== requestId) return // 不是最新的，丢弃
  render(res)
}
```

### 追问链

- 悲观锁和乐观锁分别适合什么场景？
- 自旋锁和互斥锁的区别？什么时候用自旋锁？
- 分布式锁怎么实现？Redis SETNX 有什么坑？
- `navigator.locks` 是怎么工作的？

### 易错点

- 认为前端不需要任何并发控制——只要涉及多进程/多 worker/外部资源就需要
- 自旋锁用在长临界区，反而浪费 CPU
- 误以为乐观锁不会失败——它只是把冲突推迟到提交时检测

### 知识延伸

- CAS（Compare And Swap）原子操作，乐观锁的底层实现
- `Atomics` API + `SharedArrayBuffer`：浏览器/Node 中真正的原子操作
- 数据库的间隙锁、意向锁、行锁、表锁

## Q2: 死锁是什么

### 问题

死锁是什么？产生死锁需要满足哪些条件？怎么避免？

### 参考答案

**死锁（Deadlock）：** 两个或多个执行单元互相持有对方需要的资源，又都在等对方释放，结果谁都走不了，永久卡死。

**经典场景：**
- 线程 A 拿着锁 1，想要锁 2
- 线程 B 拿着锁 2，想要锁 1
- 互不松手 → 永久阻塞

**死锁产生的四个必要条件**（必须同时满足，缺一不可）：

1. **互斥** — 资源同一时间只能被一个执行单元占用
2. **持有并等待** — 拿着已有资源的同时还在等其他资源
3. **不可剥夺** — 资源不能被强制抢走，只能由持有者主动释放
4. **循环等待** — 形成一个等待环（A 等 B，B 等 C，C 等 A）

**避免死锁——破坏任一条件即可：**

| 方法 | 破坏的条件 |
|------|-----------|
| 顺序加锁（所有线程按统一顺序申请锁） | 循环等待 |
| 一次性申请所需全部资源 | 持有并等待 |
| 超时机制（`tryLock` 超时则释放已有锁回退） | 持有并等待 |
| 死锁检测 + 强制回滚 | 不可剥夺 |

实际开发中**顺序加锁** + **缩短临界区**是最常用的组合。

**前端场景里的"类死锁"：**

- **Promise 互等**：A 的 resolve 依赖 B 完成，B 的 resolve 依赖 A 完成 → 永远 pending

  ```js
  let resolveA, resolveB
  const promiseA = new Promise(r => resolveA = r)
  const promiseB = new Promise(r => resolveB = r)

  async function taskA() { await promiseB; resolveA() }
  async function taskB() { await promiseA; resolveB() }

  taskA(); taskB() // 永远 pending
  ```

- **数据库行锁**：两个事务交叉更新同样的行，MySQL 自己会检测并回滚一个，报 `Deadlock found when trying to get lock`
- **async/await 互等**：两个异步函数互相 await 对方结果

### 追问链

- 死锁的四个必要条件具体怎么破坏？
- MySQL 是怎么检测死锁的？（等待图算法）
- 活锁（Livelock）和饥饿（Starvation）和死锁有什么区别？

### 易错点

- 只记一两个必要条件，忘记四个要同时满足
- 把"超时"当万能药——超时只能缓解持有并等待，不能根治
- 误以为单线程语言（如 JS）不会死锁——Promise/async 间互等也是死锁

### 知识延伸

- 银行家算法（Banker's Algorithm）：死锁避免的经典算法
- 活锁：双方都"礼让"反而都进不去（持续重试但不前进）
- 哲学家就餐问题、生产者消费者问题

## Q3: 顺序加锁（Lock Ordering）

### 问题

什么是顺序加锁？它是怎么解决死锁的？前端有对应的思路吗？

### 参考答案

**顺序加锁：** 给所有锁定一个**全局排序**，所有执行单元严格按这个顺序申请锁，从根本上消除"循环等待"这个死锁条件。

**死锁版本（顺序不一致）：**

```js
// 操作 A：转账 user1 → user2
lock(user1)  // 先锁 user1
lock(user2)  // 再锁 user2

// 操作 B：转账 user2 → user1
lock(user2)  // 先锁 user2  ← 顺序反了
lock(user1)  // 再锁 user1
```

时序：
```
时刻 1：A 拿到 user1
时刻 2：B 拿到 user2
时刻 3：A 想拿 user2 → 被 B 占着
时刻 4：B 想拿 user1 → 被 A 占着
→ 永久卡死
```

**顺序加锁版本：**

```js
// 不管谁转给谁，都按账户 ID 从小到大加锁
function transfer(fromId, toId) {
  const [first, second] = fromId < toId
    ? [fromId, toId]
    : [toId, fromId]

  lock(first)
  lock(second)
  // ... 转账
  unlock(second)
  unlock(first)
}
```

无论是 1→2 还是 2→1，**两个线程都会先抢 user1 的锁**，谁抢到谁先走，根本不存在交叉等待。

**核心思想：**
- 给所有锁定义一个**全局排序**（按 ID、内存地址、名称都行）
- 所有执行单元申请锁时严格按这个顺序
- 后申请的锁如果序号比已持有的小，必须先释放再重新申请

**类比理解（餐厅拼桌）：**
- **死锁**：你抢到 1 号桌想拼 2 号桌，朋友抢到 2 号桌想拼 1 号桌，僵住
- **顺序加锁**：餐厅规定永远从小号桌开始坐，两人都先去 1 号桌登记，不会撞上

**前端对应的思路——"统一执行顺序"：**

虽然前端不写真锁，但"统一顺序解决并发冲突"思想随处可见：

| 场景 | 对应顺序约定 |
|------|-------------|
| 多个 Modal 弹出冲突 | 按优先级排序（error > warning > info） |
| 批量加载资源（图片/字体/配置） | 所有模块按同一顺序申请 |
| Redux action 处理 | 串行执行队列，不并发 |
| Vue 更新队列（nextTick） | 异步合并按顺序 flush |
| Promise 链式串行 | `queue.then(...).then(...)` |

### 追问链

- 顺序加锁不能解决所有死锁场景吗？什么情况会失效？
- 如果锁的全局排序不存在（动态生成的锁）怎么办？
- 数据库里多表更新如何应用顺序加锁？

### 易错点

- 只在部分代码遵守顺序，团队里有人破坏约定就会出现死锁
- 认为顺序加锁会牺牲性能——实际上只是改变申请顺序，不增加锁的数量
- 误以为顺序加锁能避免持有并等待——它只破坏循环等待

### 知识延伸

- 锁的层次化（Lock Hierarchy）：把锁分层级，只能按层级从高到低申请
- 一次性加锁（all-at-once locking）：另一种避免死锁的策略
- 数据库事务隔离级别与锁的关系
