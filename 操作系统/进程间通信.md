# 进程间通信

## Q1: 进程间通信（IPC）有哪些方式

### 问题

进程间通信（IPC）有哪些方式？各自的特点和适用场景是什么？Node.js 中是怎么做 IPC 的？

### 参考答案

进程有独立的地址空间，不能像线程那样直接读写共享变量，必须借助操作系统提供的机制来交换数据。常见有 **6 种**方式。

**1. 管道（Pipe）**

最古老、最简单的 IPC，本质是内核里的一段缓冲区。

- **匿名管道**：只能用于**有亲缘关系**的进程（父子、兄弟），半双工，单向
  ```bash
  ps aux | grep node    # | 就是匿名管道
  ```
- **命名管道（FIFO）**：以文件形式存在，**任意两个进程**都能用，半双工

特点：简单、字节流、容量有限（一般 64KB），写满会阻塞。

**2. 消息队列（Message Queue）**

内核维护的一个**消息链表**，进程按消息类型读写，天然带边界（不像管道是字节流）。

- 可以异步通信，发送方不用等接收方在线
- 支持按消息类型选择性读取
- 受系统限制，单条消息和队列总量都有上限

**3. 共享内存（Shared Memory）**

把同一块物理内存映射到多个进程的地址空间，**最快**的 IPC 方式（不需要数据拷贝）。

- 速度极快，适合大数据量
- 需要配合**信号量/锁**做同步，否则会有竞态
- Node.js 的 `SharedArrayBuffer` + `Atomics` 就是用户态的共享内存

**4. 信号量（Semaphore）**

不是用来传数据的，而是用来**同步和互斥**——本质是一个计数器，配合共享内存使用，控制谁能进临界区。

**5. 信号（Signal）**

一种**异步通知**机制，能传递的信息很有限（就是一个信号编号）。

```bash
kill -9  <pid>    # SIGKILL，强制终止
kill -15 <pid>    # SIGTERM，请求终止（可被捕获）
```

Node.js 里 `process.on('SIGTERM', ...)` 就是监听信号做优雅退出。

**6. Socket（套接字）**

唯一**跨机器**的 IPC，分两类：

- **网络 Socket**（TCP/UDP）：跨主机通信
- **Unix Domain Socket**：同机器内不同进程，比 TCP 快很多（不走网络协议栈）

**对比表：**

| 方式 | 跨主机 | 速度 | 数据量 | 同步性 | 典型场景 |
|------|------|------|------|------|---------|
| 管道 | ❌ | 中 | 小 | 阻塞 | shell 命令链 |
| 消息队列 | ❌ | 中 | 中 | 异步 | 解耦的模块通信 |
| 共享内存 | ❌ | **最快** | 大 | 需配合信号量 | 大数据量高频通信 |
| 信号量 | ❌ | 快 | 仅同步用 | — | 配合共享内存做互斥 |
| 信号 | ❌ | 快 | 极小 | 异步 | 进程控制（kill） |
| Socket | ✅ | 慢-中 | 大 | 阻塞/异步 | 微服务、跨机通信 |

**Node.js 中的 IPC：**

Node 主要用两种封装好的方式。

`child_process` / `cluster` —— 父子进程通过 `process.send()` / `child.on('message')` 通信：

- Windows 底层是命名管道，Linux 底层是 Unix Domain Socket
- 自动 JSON 序列化，可传 handle（句柄传递实现端口共享，cluster 多进程监听同一端口靠的就是它）

```js
// 主进程
const { fork } = require('child_process')
const child = fork('worker.js')
child.send({ type: 'task', data: [1, 2, 3] })
child.on('message', msg => console.log(msg))

// worker.js
process.on('message', msg => {
  process.send({ result: msg.data.reduce((a, b) => a + b) })
})
```

`worker_threads` —— 严格说是线程间，但也用 `MessagePort` + `postMessage` 通信，并支持 `SharedArrayBuffer` 做真正的共享内存。

**选型思路：**

- 同机、父子进程、简单数据 → **管道 / Node IPC channel**
- 同机、高频大数据 → **共享内存**
- 同机、解耦异步 → **消息队列**（或上层的 Redis / Kafka）
- 跨机器 → **Socket**（或 HTTP / gRPC / MQ 等基于 Socket 的上层协议）
- 仅控制信号 → **Signal**

### 追问链

- 共享内存为什么最快？数据拷贝发生在哪些 IPC 方式里？
- cluster 模块多个 worker 怎么共享同一个端口？（句柄传递 + 内核负载均衡）
- Unix Domain Socket 和 TCP loopback（127.0.0.1）相比有什么优势？
- `SharedArrayBuffer` 在浏览器里为什么默认被禁用？（Spectre 攻击）
- Node.js 的 `process.send()` 在 Windows 和 Linux 底层分别用什么实现？

### 易错点

- 把"管道"和"消息队列"混为一谈——管道是字节流没有边界，消息队列是带类型的离散消息
- 误以为共享内存可以"一用就快"——必须配合信号量同步，否则数据会乱
- 认为 Socket 只能跨机器——Unix Domain Socket 是同机的高效 IPC，Docker、Nginx 大量使用
- 把信号当数据通道——信号只能传一个编号，不能携带业务数据
- 误以为 Node.js 的 `worker_threads` 就是 IPC——它是线程间通信，进程内地址空间共享

### 知识延伸

- mmap：内存映射文件，是共享内存的另一种实现方式
- gRPC、HTTP、消息中间件（Kafka、RabbitMQ）都是基于 Socket 的上层 IPC 协议
- Binder：Android 系统专用的 IPC 机制，比传统 Linux IPC 更高效
- D-Bus：Linux 桌面环境常用的 IPC 总线
- 浏览器多进程架构中的 IPC：Chromium 用 Mojo，进程间走命名管道 + 共享内存
