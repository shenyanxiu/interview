# 进程与线程

## Q1: 进程和线程的区别

### 问题

进程和线程的区别是什么？

### 参考答案

**基本概念：**
- **进程** 是操作系统资源分配的基本单位，是程序的一次执行实例，拥有独立的内存空间、文件描述符等资源
- **线程** 是 CPU 调度的基本单位，是进程中的一个执行流，同一进程内的线程共享进程的内存空间和资源

**核心区别：**

| 维度 | 进程 | 线程 |
|------|------|------|
| 资源 | 独立的地址空间、文件描述符等 | 共享所属进程的资源，拥有独立的栈和寄存器 |
| 开销 | 创建/销毁/切换开销大 | 创建/销毁/切换开销小 |
| 通信 | 需要 IPC（管道、消息队列、共享内存等） | 可直接读写共享变量，需注意同步 |
| 隔离性 | 一个进程崩溃不影响其他进程 | 一个线程崩溃可能导致整个进程终止 |
| 并发性 | 进程间并发，适合多核并行 | 线程间并发，更轻量 |

**内存模型：**
- 同一进程内的线程共享代码段、数据段和堆
- 各线程拥有独立的栈空间

**与 Node.js 的关系：**
- 主线程（JS 执行）是单线程的
- 底层 libuv 维护线程池（默认 4 个线程）处理文件 I/O 等阻塞操作
- `worker_threads` 可创建工作线程，适合 CPU 密集型任务
- `child_process` / `cluster` 可创建子进程，利用多核 CPU

### 追问链

- Node.js 是单线程的吗？底层呢？
- 进程间通信（IPC）有哪些方式？
- 什么场景选多进程，什么场景选多线程？

### 易错点

- 误认为"线程拥有独立内存空间"——线程共享进程的堆和数据段，只有栈是独立的
- 混淆"单线程"和"单进程"——Node.js 是单线程但可以是多进程（cluster）

### 知识延伸

- 协程（Coroutine）：比线程更轻量的用户态调度单位
- Go 的 Goroutine、Python 的 asyncio
- 浏览器的多进程架构（见 Q2）

## Q2: Chrome 浏览器是多进程还是多线程架构

### 问题

Chrome 浏览器采用的是多进程还是多线程？为什么这样设计？

### 参考答案

Chrome 采用**多进程 + 进程内多线程**的混合架构。

**主要进程：**

| 进程 | 职责 |
|------|------|
| Browser 进程 | 主进程，负责地址栏、书签、网络请求、文件访问等 |
| Renderer 进程 | 每个标签页（或同站点组）一个，负责页面渲染、JS 执行、DOM 解析 |
| GPU 进程 | 统一处理所有标签页的 GPU 绘制任务 |
| Plugin 进程 | 每种插件一个进程 |
| Extension 进程 | 每个浏览器扩展一个独立进程 |
| Utility 进程 | 处理网络服务、音频服务等 |

**为什么选择多进程：**
1. **稳定性** — 一个标签页崩溃不会拖垮整个浏览器
2. **安全性** — 渲染进程运行在沙箱中，被攻破也无法访问系统资源
3. **隔离性** — 不同站点互不干扰，避免内存泄漏相互影响

**Renderer 进程内部的多线程：**
- 主线程（Main Thread）— JS 执行、DOM 解析、样式计算、布局
- 合成线程（Compositor）— 图层合成、处理滚动
- 光栅化线程（Raster）— 将图层绘制为位图
- Worker 线程 — Web Worker / Service Worker
- I/O 线程 — 处理 IPC 通信

### 追问链

- 同站点的多个标签页是共享一个 Renderer 进程吗？（Site Isolation）
- 渲染进程的沙箱机制是怎么实现的？
- 为什么 GPU 进程只有一个而不是每个标签页一个？

### 易错点

- 简单回答"多进程"不够完整——应该说明是多进程 + 进程内多线程的混合架构
- 误认为每个标签页一定是独立进程——同站点标签页可能共享 Renderer 进程（Site Isolation 策略）

### 知识延伸

- Site Isolation：Chrome 67+ 默认开启，同源策略在进程级别隔离
- 进程模型的内存代价：每个进程有独立内存开销，标签页过多时内存占用高
- Chromium 的 Servicification：将 Browser 进程的功能拆分为独立服务，按设备性能动态合并或拆分进程
